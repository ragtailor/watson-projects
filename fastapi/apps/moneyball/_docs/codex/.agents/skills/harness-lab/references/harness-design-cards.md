# 하네스 설계 카드

이 문서는 `harness-lab`이 사용자의 일을 Codex 하네스 구조로 바꾸기 전에 목표, 자료, 검증, 승인 지점을 차분히 정리할 때 사용한다. 초보자에게는 파일 구조보다 먼저 이 카드를 보여준다.

## 목차

1. 언제 읽을까
2. 하네스 한 문장 정의
3. 숙련도 적응
4. 하네스 7요소
5. 하네스 성숙도 5단계
6. 하네스 적합성 게이트
7. 한 장 설계 카드
8. 하네스 실습 키트 7장
9. 하네스 두께 정하기
10. 청사진 입력 프롬프트
11. 청사진에 반드시 반영할 것

## 언제 읽을까

- 사용자가 "내 업무를 하네스로 바꿔줘"라고 했지만 입력이 아직 막연할 때
- 하네스 성숙도, 하네스 두께, 설계 카드, 실습 키트가 필요할 때
- 청사진을 만들기 전에 사용자의 일을 7요소로 정리해야 할 때
- 사용자가 "에이전트와 스킬을 만들어줘"라고 했지만 최종 산출물과 검증 기준이 불분명할 때

## 하네스 한 문장 정의

하네스는 AI가 덜 추측하고, 더 안전하게, 더 반복 가능하게 일하도록 만드는 작업 환경이다.

좋은 하네스는 AI에게 모든 것을 즉석에서 판단하게 하지 않는다. 목표는 목표로, 자료는 자료로, 반복 절차는 Skill로, 역할은 Agent로, 승인과 검증은 Orchestrator와 체크리스트로 바깥에 꺼내 둔다.

## 숙련도 적응

같은 하네스라도 사용자의 숙련도에 따라 설명의 밀도를 바꾼다. 대화의 단서(쓰는 용어, 질문 수준)로 수준을 가늠하고 톤을 조절한다. 결과물 구조는 같고 설명의 두께만 달라진다.

| 수준 | 단서 | 설명 방식 |
| --- | --- | --- |
| 초보 | 하네스·Agent·Skill 용어가 낯설다, "멋지게/알아서" 같은 막연한 표현 | 비유 먼저, 아래 보편 인테이크 5문항으로 빈칸을 채운 뒤 청사진 |
| 실무자 | 산출물·역할·검토를 구체적으로 말한다 | 청사진과 핵심 결정 위주. 역할표·산출물 계약을 바로 보여준다 |
| 전문가 | model·sandbox·승인 정책·평가 용어를 직접 쓴다 | 설계 카드와 스펙 위주로 간결하게. 트레이드오프만 짚는다 |

초보 수준일 때는 업종·기술과 무관하게 동작하는 **보편 인테이크 5문항**을 먼저 채운다.

1. 무엇을 만들면 성공인가? (최종 산출물)
2. 누가 읽거나 쓰는가? (독자·사용자)
3. 어떤 자료를 먼저 줄 수 있는가? (입력)
4. 무엇이 틀리면 위험한가? (위험·승인 지점)
5. 한 번 쓰고 끝인가, 반복하는가? (반복성)

이 다섯 답이 산출물 유형, 역할 구성, 두께와 승인 게이트를 거의 결정한다. 수준을 점수처럼 말하지 않고 부족한 칸만 "확인 필요"로 표시한 뒤 진행한다.

## 하네스 7요소

| 요소 | 쉬운 말 | 확인 질문 | Codex 산출물 |
| --- | --- | --- | --- |
| 목표 | 무엇을 만들지 정하기 | 최종 결과물은 무엇인가? | 목표 한 문장, 완료 기준 |
| 컨텍스트 | AI가 봐야 할 자료 | 어떤 문서와 정보를 먼저 줄 것인가? | 참고 자료 목록, 입력 폴더 |
| 도구 | AI가 사용할 손과 발 | 어떤 도구가 필요하고 어디까지 허용할 것인가? | Agent 지시, 권한 안내 |
| 중간 산출물 | 바로 최종으로 가지 않는 중간 결과 | 초안 전에 무엇을 먼저 확인할 것인가? | `artifacts/`의 요약표, 매트릭스, 검토표 |
| 검증 | 좋은 결과의 기준 | 무엇을 만족해야 괜찮은 결과인가? | 체크리스트, 테스트 프롬프트, 루브릭 |
| 권한과 승인 | 사람이 책임질 지점 | 발송, 게시, 제출, 삭제 전에 누가 확인하는가? | Orchestrator의 승인 조건 |
| 기록과 개선 | 다음번을 위한 기억 | 무엇을 저장하고 다음번에 고칠 것인가? | `AGENTS.md` 변경 이력, 개선 기록 |

처음부터 7요소를 모두 완벽히 채울 필요는 없다. 빈칸은 AI가 지금까지 추측으로 메우던 부분이므로, 오히려 좋은 설계 단서가 된다.

## 하네스 성숙도 5단계

| 단계 | 이름 | 상태 | 다음 보강 |
| --- | --- | --- | --- |
| 1단계 | 요청만 있는 상태 | 프롬프트는 있지만 기준과 검증이 없다 | 목표, 대상, 참고 자료, 금지 조건을 적는다 |
| 2단계 | 목표와 자료가 있는 상태 | 무엇을 해야 하는지와 참고 자료가 있다 | 중간 산출물과 질문 목록을 둔다 |
| 3단계 | 중간 산출물과 검증이 있는 상태 | 초안 전에 기준표와 검토표가 있다 | 실패 처리와 사람 승인 조건을 정한다 |
| 4단계 | 실패 처리와 인수인계가 있는 상태 | 재시도, 기록, 다음 작업 인계가 가능하다 | 실행 기록과 개선 루프를 둔다 |
| 5단계 | 운영과 개선 루프가 있는 상태 | 승인, 메모리, 지표, 개선 루프가 함께 돈다 | 오래된 규칙과 메모리를 정리한다 |

진단할 때는 점수 매기듯 말하지 않는다. "지금은 2단계에 가까워서, 중간 산출물 하나와 검증 기준 세 개를 붙이면 바로 좋아집니다"처럼 다음 행동으로 연결한다.

## 하네스 적합성 게이트

하네스를 만들기 전에 "이 업무가 하네스로 만들 가치가 있는가"를 먼저 판정한다. 만들기 쉽다고 다 만들면, 일회성·고변동·전문가 업무에서는 오히려 손이 더 가거나 느려진다. 실제로 숙련자가 성숙한 작업에 AI를 붙였을 때 본인은 빨라졌다고 느끼지만 측정해 보면 더 느려진 사례가 있다. 체감과 실제가 다를 수 있으므로, 진입부에서 아래 6차원을 보고 이득 쪽이 분명할 때만 두껍게 만든다.

| 차원 | 하네스가 이득 | 손해(얇게/생략) | 확인 질문 |
| --- | --- | --- | --- |
| 반복성 | 같은 절차를 자주 반복 | 일회성·매번 다른 일 | 이 일을 앞으로도 반복하는가? |
| 구조 안정성 | 흐름을 미리 그릴 수 있다 | 매번 즉흥적으로 달라진다 | 단계와 분기를 사전에 매핑할 수 있는가? |
| 검증 가능성 | 정답·체크리스트로 값싸게 검증 | 검증이 생성만큼 어렵다 | 좋은 결과인지 객관적으로 가릴 수 있는가? |
| 외부 신호 | 테스트·기준·사람 승인이 있다 | 순수 자기평가에만 의존 | 검증을 무엇에 기댈 수 있는가? |
| 숙련도 | 저숙련·표준화 업무에 절차가 도움 | 고숙련·고맥락 전문가 업무 | 누가 하든 같은 절차가 도움이 되는가? |
| 체인 길이 | 짧고 검증 게이트가 있다 | 긴 다단계 자율 체인 | 단계가 길어 오류가 누적되지 않는가? |

판정은 점수가 아니라 방향이다. 이득 쪽이 분명하면 두껍게, 손해 신호가 많으면 단일 흐름·체크리스트로 얇게 가거나 하네스를 만들지 않는다. 가치가 애매하면 with/without 비교(`light`)로 먼저 확인한 뒤 결정한다. 효율은 "에이전트를 많이 둬서"가 아니라 구조화·검증·승인 게이트에서 나오므로, 멀티에이전트 자체를 목표로 삼지 않는다.

## 한 장 설계 카드

청사진 전에 아래 항목을 채운다. 모든 항목이 비어 있으면 먼저 질문하고, 절반 이상 채워져 있으면 남은 빈칸을 "확인 필요"로 표시한 뒤 청사진을 만들 수 있다.

| 항목 | 적을 내용 |
| --- | --- |
| 1. 반복해서 하는 일 | 어떤 일을 하네스로 만들고 싶은가? |
| 2. 흔들리는 이유 | 결과가 매번 달라지는 이유는 무엇인가? |
| 3. 최종 결과물 | 마지막에 어떤 형태의 결과가 나와야 하는가? |
| 4. 먼저 줄 자료 | 문서, 메모, 링크, 기준표, 예시 중 무엇을 줄 것인가? |
| 5. 먼저 물어볼 질문 | 정보가 부족할 때 AI가 무엇을 물어봐야 하는가? |
| 6. 중간 산출물 | 최종 결과 전에 어떤 표, 목록, 초안, 체크리스트가 필요한가? |
| 7. 검증 기준 | 무엇을 만족해야 좋은 결과라고 볼 수 있는가? |
| 8. 실패 처리 | 다시 시도할 것인가, 질문할 것인가, 멈출 것인가? |
| 9. 사람 승인 지점 | 어떤 순간에는 반드시 사람이 확인해야 하는가? |
| 10. 다음 기록 | 다음 작업을 위해 무엇을 저장하거나 기억해야 하는가? |
| 11. 핵심 결정(ADR) | 큰 구조 결정은 무엇이고, 버린 대안·가정·트레이드오프·재검토 조건은 무엇인가? (작성 형식은 `failure-maintenance.md`) |

## 하네스 실습 키트 7장

사용자가 워크숍이나 교육 흐름을 원하면 아래 카드 묶음으로 안내한다.

| 키트 | 이름 | 목적 |
| --- | --- | --- |
| 1 | 내 업무 선택 카드 | 하네스로 만들 반복 업무를 하나 고른다 |
| 2 | 컨텍스트 수집 카드 | AI에게 먼저 줄 자료를 정리한다 |
| 3 | 질문 카드 | AI가 추측하지 말고 물어봐야 할 질문을 정한다 |
| 4 | 중간 산출물 카드 | 바로 최종 결과로 가지 않도록 중간 결과를 만든다 |
| 5 | 검증 기준 카드 | 좋은 결과를 판단할 기준을 만든다 |
| 6 | 실패 처리와 승인 카드 | 재시도, 중단, 사람 확인 지점을 정한다 |
| 7 | 다음 개선 카드 | 다음번에 더 잘하기 위한 기록을 남긴다 |

## 하네스 두께 정하기

하네스 두께는 AI에게 자유를 더 줄지, 체크리스트와 검증으로 더 촘촘히 잡을지 정하는 판단이다.

| 상황 | 적당한 두께 | 이유 |
| --- | --- | --- |
| 아이디어 브레인스토밍 | 얇은 하네스 | 자유로운 탐색이 더 중요하다 |
| 개인 메모 정리 | 얇은 하네스 | 실패 비용이 낮다 |
| 팀 회의록 공유 | 중간 하네스 | 누락과 오해를 줄여야 한다 |
| 고객 답변 | 중간에서 두꺼운 하네스 | 외부 영향과 브랜드 위험이 있다 |
| 사업계획서 제출 | 두꺼운 하네스 | 평가 기준, 증빙, 제출 책임이 있다 |
| 결제, 삭제, 배포, 법적 판단 | 매우 두꺼운 하네스 | 되돌리기 어렵고 책임이 크다 |

두꺼운 하네스가 항상 좋은 것은 아니다. 위험이 낮고 창의성이 중요한 일에 너무 두꺼운 흐름을 붙이면 속도와 재미가 떨어진다. 반대로 외부 발송, 제출, 결제, 삭제에는 사람 승인과 검증을 반드시 둔다.

두께는 위험도이고, 두께가 곧 켜지는 검증과 사람 승인의 양이다. 한 축으로 묶어 자동으로 맞춘다.

| 두께 | 켜지는 rigor |
| --- | --- |
| 얇음 | frontmatter 위생 + 산출물 파일 + 기본 테스트 |
| 중간 | 위 + 검토 역할 + 발송 전 사람 승인 + A/B·near-miss 검증 |
| 두꺼움 | 위 + 증거 기반 검증 + 사람 승인 강화 + 미검증 영역 명시 |

특정 업종에 맞춘 고정 구성을 미리 넣지 않는다. 사용자가 요청한 일에서 위험 신호(외부 발송·제출·결제·삭제·개인정보·법적 표현)를 읽어 두께와 사람 승인 지점을 그때 정한다.

## 청사진 입력 프롬프트

사용자가 자신의 업무를 정리해 오도록 도울 때 아래 형식을 제안한다.

```text
$harness-lab 아래 설계 카드를 바탕으로 Codex에서 사용할 하네스를 설계해줘.

1. 반복 업무:

2. 먼저 줄 자료:

3. AI가 먼저 물어봐야 할 질문:

4. 중간 산출물:

5. 검증 기준:

6. 실패 처리와 사람 승인:

7. 다음 개선 기록:

먼저 청사진을 보여주고, 내가 승인하면 실제로 사용할 수 있는 agents와 skills를 만들어줘.
```

## 청사진에 반드시 반영할 것

- 최종 결과물과 중간 산출물을 구분한다.
- AI가 추측하면 안 되는 부분을 "먼저 질문"으로 빼둔다.
- Agent는 필요한 만큼만 만든다.
- Skill은 반복 절차가 있는 역할에만 만든다.
- Orchestrator에는 단계별 산출물과 사람 승인 조건을 둔다.
- 테스트 프롬프트에는 정상, 애매함, 실패 위험을 모두 포함한다.
