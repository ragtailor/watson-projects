## 언제 읽을까

- 사용자가 "에이전트와 스킬을 만들어줘"라고 요청할 때
- Agent를 몇 개 만들지 판단해야 할 때
- Skill description, workflow, output format, edge case를 설계해야 할 때
- `.codex/agents/{name}.toml`에 어떤 지시를 넣을지 정해야 할 때
- Orchestrator와 일반 Skill의 경계를 정해야 할 때

## Agent, Skill, Orchestrator의 차이

| 요소 | 쉬운 비유 | 담는 내용 | Codex 위치 |
| --- | --- | --- | --- |
| Agent | 일을 맡은 팀원 | 역할, 책임, 입력, 출력, 도구 범위, 하지 말아야 할 일 | `.codex/agents/{name}.toml` |
| Skill | 업무 매뉴얼 | 트리거, 절차, 출력 형식, 품질 기준, 예외 처리 | `.agents/skills/{name}/SKILL.md` |
| Orchestrator | 팀장과 진행표 | 순서, 담당자, 중간 산출물, 실패 처리, 승인 조건 | `.agents/skills/{harness-name}-orchestrator/SKILL.md` |

좋은 하네스는 담당자 카드와 업무 매뉴얼을 분리한다. Agent 파일에 모든 절차를 길게 넣으면 재사용이 어렵고, Skill에 모든 역할 정보를 넣으면 누가 책임지는지 흐려진다.

## Agent를 분리하기 전 질문

Agent는 많을수록 좋은 것이 아니다. 아래 질문 중 두 개 이상이 "예"일 때 분리를 검토한다.

| 질문 | 분리해도 좋은 신호 | 분리하지 않는 편이 나은 신호 |
| --- | --- | --- |
| 문맥 경계가 분명한가? | 서로 다른 자료를 봐도 된다 | 같은 정보를 계속 이어서 봐야 한다 |
| 전문성이 다른가? | 조사, 재무, 편집, 법무처럼 판단 기준이 다르다 | 같은 산출물을 조금씩 나눠 쓰는 정도다 |
| 입력과 출력이 독립적인가? | 결과만 넘겨도 다음 단계가 쓸 수 있다 | 중간 판단을 계속 공유해야 한다 |
| 반복 재사용할 책임인가? | 다른 하네스에서도 자주 부를 역할이다 | 이번 한 번만 필요한 작은 단계다 |
| 빠지면 위험한가? | 검증, 승인, 근거 확인처럼 품질이나 안전에 영향이 크다 | Orchestrator 체크리스트로 충분하다 |

처음 실습은 보통 3-4개 Agent가 적당하다. 긴 문서 작성, 프로젝트 운영, 제출 준비처럼 흐름이 길면 5-7개까지 가능하지만, 각 Agent가 왜 별도 역할인지 설명해야 한다.

## 문맥 경계로 나누기

역할 이름만 보고 나누지 않는다. 중요한 기준은 어떤 정보를 함께 봐야 하는지다.

- 같은 정보를 계속 이어서 해석해야 하면 한 Agent가 맡는다.
- 독립적으로 조사하고 결과만 합치면 나눌 수 있다.
- 중간 발견이 다른 작업 방향을 바꾸면 Orchestrator가 파일 산출물과 전달 흐름을 명확히 둔다.
- 여러 Agent가 같은 파일을 동시에 고쳐야 하면 병렬 수정보다 순차 흐름이나 검토 역할을 우선한다.

일상 비유로 보면, 우유 하나 사러 가는 데 가족 회의를 열 필요는 없다. 하지만 이사 준비처럼 날짜, 비용, 포장, 주소 이전이 서로 영향을 주면 역할과 조율 구조가 필요하다.

## 커스텀 Agent를 만들기 좋은 경우

- 자료 조사, 코드 탐색, 문서 요약처럼 독립 작업을 맡길 때
- 보안, 성능, 문체, 재무처럼 관점이 뚜렷한 검토가 필요할 때
- 같은 기준으로 반복 호출할 전문 담당자가 필요할 때
- 특정 역할의 말투, 판단 기준, 금지 행동을 프로젝트 안에 고정하고 싶을 때

## 커스텀 Agent를 피하는 편이 나은 경우

- 한 사람이 차분히 처리하는 편이 더 빠른 단순 작업
- 한 번 쓰고 끝나는 아주 작은 단계
- 여러 Agent가 같은 파일을 동시에 수정해야 하는 작업
- 결과 품질을 측정할 기준이 없어 비용만 늘어날 수 있는 작업

## Agent TOML 필수 요소

```toml
name = "example_agent"
description = "이 Agent가 언제 호출되어야 하는지 구체적으로 쓴다."

developer_instructions = """
당신은 {역할 이름}입니다.

책임:
- 이 Agent가 맡는 일을 적는다.

입력:
- 어떤 자료와 조건을 받아야 하는지 적는다.

출력:
- 다음 단계가 바로 사용할 수 있는 산출물 형식을 적는다.

작업 방식:
1. 목적과 산출물을 확인한다.
2. 필요한 자료를 읽는다.
3. 근거와 추측을 구분한다.
4. 정해진 형식으로 지정 경로에 결과를 저장한다. 저장에 실패하면 재시도하고, 그래도 안 되면 차단을 보고한다.

하지 말아야 할 일:
- 역할 밖의 결정을 확정하지 않는다.
- 확인하지 않은 사실을 단정하지 않는다.
- 산출물을 지정 경로에 저장하지 않고 본문 요약으로 대체하지 않는다.
- 외부 발송, 제출, 삭제, 결제처럼 책임이 생기는 행동을 사람 승인 없이 완료하지 않는다. 승인이 필요하면 멈추고 사용자에게 승인을 요청하며, 묻지 않고 통과하지 않는다.
"""
```

필요할 때만 `model`, `model_reasoning_effort`, `sandbox_mode`, MCP 설정을 제안한다. 첫 실습에서는 실행 설정 파일을 과하게 늘리지 않는다.

## Skill의 5가지 구성요소

| 구성요소 | 역할 | 작성 팁 |
| --- | --- | --- |
| YAML frontmatter | 이름과 트리거 | description은 소개문이 아니라 출입 조건이다 |
| Overview | 목적과 결과 기준 | 한 단락으로 짧게 쓴다 |
| Workflow | 작업 순서 | 번호를 붙이고 한 단계에 한 행동만 둔다 |
| Output Format | 결과 모양 | 다음 단계가 그대로 읽을 수 있게 한다 |
| Examples and Edge Cases | 정상·예외 예시 | 부족한 정보, 쓰면 안 되는 경우를 포함한다 |

## Description 작성법

좋은 description에는 세 가지가 들어간다.

1. 이 Skill이 무엇을 하는지
2. 사용자가 어떤 말을 할 때 써야 하는지
3. 어떤 요청에는 쓰면 안 되는지

나쁜 예:

```yaml
description: 문서 작업을 도와줍니다.
```

좋은 예:

```yaml
description: >
  고객용 제안서 초안을 작성하고 검토합니다.
  사용자가 "제안서 작성", "고객 제안서", "외부 발송용 제안 문서",
  "프로젝트 제안서 초안"을 요청할 때 사용하세요.
  내부 회의 메모, 기술 사양서, 단순 맞춤법 교정에는 사용하지 마세요.
```

description이 약하면 Skill이 실행되지 않는다. description이 너무 넓으면 엉뚱한 요청을 가로챈다.

## frontmatter 하드 제약

Skill의 `name`과 `description`은 형식 제약이 있다. 생성된 Skill이 안정적으로 로딩·트리거되려면 이를 지킨다.

| 필드 | 제약 |
| --- | --- |
| `name` | 최대 64자, 소문자·숫자·하이픈만, 예약어 `claude`·`anthropic` 사용 금지 |
| `description` | 비어 있지 않게, 최대 1024자, **무엇을 + 언제** 모두 포함 |

네이밍은 동작이 보이는 구체적 이름(`writing-reports`)을 쓰고 `helper`·`utils`·`documents`처럼 모호한 이름은 피한다. `description`은 시스템 프롬프트에 주입되므로 **3인칭**으로 쓴다. "나는 도와줄 수 있다"가 아니라 "제안서를 작성하고 검토한다"처럼 행동을 서술한다. (Codex Agent의 `.codex/agents/{name}.toml` `name`·`description`도 같은 기준으로 구체적으로 쓴다.)

## Skill 본문 작성 원칙

- 에세이가 아니라 업무 지시서로 쓴다.
- 중요한 규칙에는 이유를 붙인다.
- 결과 형식과 금지 행동을 명확히 둔다.
- 길어지면 `references/`로 분리하고, 본문에는 언제 읽을지 적는다.
- 반복되고 결정적인 절차는 사람이 기억하게 하지 말고 체크리스트나 스크립트 후보로 둔다.
- 정상 입력, 최소 입력, 예외 입력, 부정 입력을 하나씩 생각한다.

## Progressive Disclosure와 자유도

로딩은 3단계로 일어나고 각 단계의 컨텍스트 비용이 다르다. 본문은 **500줄 미만**으로 유지하고, 넘으면 세부를 `references/`로 빼고 포인터만 남긴다. 참조는 한 단계 깊이로 두고, 100줄 넘는 참조에는 목차를 둔다.

| 폴더 | 용도 | 읽나/실행하나 |
| --- | --- | --- |
| `references/` | 필요할 때 읽어서 판단에 쓰는 문서 | 읽음 |
| `scripts/` | 실행하는 결정적 코드. 출력만 컨텍스트로 | 실행 |
| `assets/` | 산출물에 들어가는 재료(템플릿, 보일러플레이트, 이미지) | 결과물에 채워넣음 |

assets는 에이전트가 직접 붙이는 것이 아니라 **Skill이 참조**하고, 그 Skill을 쓰는 에이전트가 거쳐 사용한다.

지시 강도는 작업이 깨지기 쉬운 정도에 맞춘다. 정답이 여럿이면 텍스트로 방향만 주고(높은 자유도), 깨지기 쉽고 일관성이 중요하면 "이 절차를 그대로 실행한다"처럼 좁힌다(낮은 자유도). 한 강도로 통일하지 않는다.

## Orchestrator 설계 원칙

Orchestrator는 모든 일을 직접 하는 존재가 아니다. 일을 나누고, 산출물을 이어받고, 실패와 승인 지점을 관리하는 입구다.

반드시 포함할 것:

- 입력 확인과 부족한 정보 질문
- 단계별 담당 Agent 또는 Skill
- 단계별 중간 산출물 파일
- 다음 단계로 넘길 정보
- 실패했을 때 재시도, 질문, 중단 조건
- 사람이 승인해야 하는 지점
- 정상, 애매함, 실패 위험 테스트 프롬프트
- 직접 호출 예시: `$harness-name-orchestrator ...`

Orchestrator Skill 이름은 반드시 `{harness-name}-orchestrator`로 끝난다.

## 위임 4종 세트와 멀티에이전트 비용

Task를 맡길 때 모호한 한 줄("이거 조사해줘")은 중복·누락·방향 어긋남의 가장 큰 원인이다. 각 위임에는 네 가지를 담는다.

| 요소 | 담을 내용 |
| --- | --- |
| 목표 | 이 Task가 끝나면 무엇이 있어야 하는가 |
| 출력 형식 | 어떤 파일에 어떤 구조로 남기는가 |
| 도구·출처 | 어떤 도구와 자료를 쓰고 어디는 보지 말아야 하는가 |
| 경계 | 어디까지가 이 Task의 책임이고 어디부터는 다른 역할인가 |

파일 산출물을 맡기는 Task의 출력 형식에는 저장 계약을 함께 적는다: 지정 경로에 반드시 저장하고, 저장에 실패하면 재시도한 뒤 그래도 안 되면 차단을 보고한다. **저장에 성공한 척 본문 요약으로 대체하지 않는다.** Codex Agent는 세션 도구를 상속하므로 별도 권한 부여는 필요 없지만, 같은 하네스를 Claude Code로 옮기면 그 Agent가 `Write` 권한을 갖췄는지(검토·QA도 자기 판정 파일을 쓰므로 포함) 확인한다.

멀티에이전트는 공짜가 아니다. 여러 에이전트가 각자 컨텍스트를 돌리면 단일 흐름보다 토큰을 훨씬 많이 쓴다. 단순한 것에서 시작해, 가치가 높고 폭넓고 병렬화되는 작업에만 여러 Agent를 쓴다. 모델은 역할에 맞춰 섞는다. 통합·위험 판단을 맡는 리드에는 강한 모델을, 병렬 조사·작성 워커에는 중간 모델을, 단순 분류·검색에는 가벼운 모델을 둔다(`.codex/agents`의 `model`, `model_reasoning_effort`). 모든 Agent에 같은 최고 모델을 주는 것은 보통 과설계다.

## 생성자-평가자 분리

긴 작업에서 가장 흔한 실패는 결과물을 만든 Agent가 자기 결과를 스스로 합격으로 판정하는 것이다. 모델은 자기 산출물을 과대평가하는 경향(자기평가 편향)이 있어, 생성과 평가를 한 역할에 두면 검증이 형식만 남는다. 그래서 위험도가 있는 하네스에서는 만드는 역할(Generator)과 평가하는 역할(Evaluator)을 분리한다.

| 구분 | Generator | Evaluator |
| --- | --- | --- |
| 목적 | 산출물을 만든다 | 산출물이 기준을 넘는지 회의적으로 판정한다 |
| 보는 것 | 작업 맥락 전체 | 결과물과 합격 기준만 (작성 맥락은 최소화) |
| 튜닝 방향 | 충실하게 완성 | 의심을 기본값으로, 애매하면 불합격 |
| 도구 | 작성에 필요한 범위 | 읽기·검증 중심으로 좁힘 |

Evaluator를 회의적으로 튜닝하는 방법:

1. 합격·불합격 판정 예시를 2-3개씩 본문에 넣어 기준을 고정한다. "이 정도면 통과", "이건 왜 탈락"을 구체적으로 보여준다.
2. 채점 기준을 항목으로 분해하고, 품질이 약한 항목에 가중치를 둔다. 모든 항목을 같은 무게로 보면 약점이 평균에 묻힌다.
3. "애매하면 불합격" 같은 기본값을 명시해 과신을 막는다.
4. Evaluator는 산출물의 해당 줄·수치·테스트 결과 같은 증거로 판정하고, "괜찮아 보인다"는 표현을 금지한다.

생성과 평가가 직접 대화로 얽히면 평가가 무뎌진다. 가능하면 Generator의 결과를 `artifacts/` 파일로 남기고, Evaluator는 그 파일과 합격 기준만 읽어 판정하게 한다. 명시적 체크포인트(파일 교환)가 실시간 대화보다 더 엄격한 검증을 만든다.

평가자는 분리만으로 충분하지 않다. 순수 자기평가는 외부 신호 없이는 오히려 품질을 떨어뜨릴 수 있고, 평가자도 자기 출력을 선호하거나 더 길거나 앞선 답을 선호하는 편향을 가진다. 그래서 Evaluator의 판정은 가능하면 외부 신호에 묶는다: 테스트 실행 결과, 체크리스트, 결함 주입 회귀 세트, 수치·인용 대조, 사람 승인. 통과까지 반복하는 검증 루프와 다수결·자기비판 패턴은 `pattern-catalog.md`의 동적 제어 패턴을 따른다.

## Skill 실패 진단

| 증상 | 먼저 고칠 곳 | 이유 |
| --- | --- | --- |
| 써야 할 때 실행되지 않는다 | `description` | 본문을 읽기 전에 description으로 선택된다 |
| 엉뚱한 요청에도 실행된다 | `description`의 제외 조건 | 부정 케이스가 부족하다 |
| 실행은 되지만 형식이 틀린다 | `SKILL.md` 본문 | 출력 형식과 예시가 약하다 |
| 같은 검사를 계속 빼먹는다 | 체크리스트 또는 검증 Skill | 기억보다 구조가 필요하다 |
| 같은 보조 코드를 매번 다시 만든다 | `scripts/` 후보 | 반복 결정 절차는 파일로 분리할 수 있다 |
| 모델 업데이트 뒤 흔들린다 | 테스트 프롬프트 세트 | 회귀 테스트가 필요하다 |
