## 패턴을 고르기 전 확인

복잡해 보인다고 바로 여러 Agent로 나누지 않는다. 먼저 아래를 확인한다.

| 질문 | 의미 |
| --- | --- |
| 한 사람이 차분히 처리해도 되는가? | 그렇다면 단일 흐름이나 Orchestrator 체크리스트로 충분할 수 있다. |
| 작업을 나눠도 문맥 손실이 작은가? | 결과만 넘겨도 다음 단계가 이해할 수 있으면 나누기 좋다. |
| 중간 발견이 다른 역할의 판단을 바꾸는가? | 그렇다면 산출물 전달과 조율 규칙이 필요하다. |
| 같은 파일이나 문서를 동시에 고쳐야 하는가? | 병렬 작업보다 순차 작성-검토 흐름이 안전하다. |

Agent는 역할 이름보다 문맥 경계로 나눈다. 같은 정보를 계속 이어서 봐야 하는 일은 붙이고, 독립적으로 조사하거나 다른 전문 기준으로 검토해야 하는 일만 분리한다.

## 1. Pipeline

한 줄로 이어지는 작업이다. 앞 단계의 결과가 다음 단계의 재료가 된다.

잘 맞는 경우:

- 순서가 분명하다.
- 중간 산출물이 차례로 쌓인다.
- 앞 단계가 틀리면 뒤 단계도 흔들린다.

주의할 점:

- 앞 단계 산출물 형식을 분명히 해야 한다.
- 마지막 검토 단계를 빼면 오류가 그대로 흘러갈 수 있다.

## 2. Fan-out/Fan-in

여러 사람이 나눠서 보고, 마지막에 한곳으로 모으는 방식이다.

잘 맞는 경우:

- 자료가 많다.
- 여러 관점이 필요하다.
- 독립적으로 나눌 수 있는 하위 작업이 있다.

주의할 점:

- 합치는 기준이 없으면 결과가 산만해진다.
- 중복 조사와 빠진 영역을 점검해야 한다.

## 3. Expert Pool

필요한 순간에 알맞은 전문가를 부르는 방식이다.

잘 맞는 경우:

- 요청마다 필요한 전문성이 다르다.
- 모든 전문가가 매번 참여할 필요는 없다.
- 판단 기준이 분야별로 다르다.

주의할 점:

- 누가 언제 호출되는지 기준이 필요하다.
- 전문가 의견이 충돌할 때 최종 판단자가 있어야 한다.

## 4. Producer-Reviewer

만드는 사람과 검토하는 사람을 분리하는 방식이다.

잘 맞는 경우:

- 품질이 중요하다.
- 초안 작성과 오류 검출의 관점이 다르다.
- 결과물을 바로 사용자에게 보여주기 전에 확인해야 한다.

주의할 점:

- 리뷰 기준이 모호하면 취향 싸움이 된다.
- 리뷰 결과를 다시 반영하는 단계가 필요하다.

## 5. Supervisor

팀장이 요청을 보고 알맞은 담당자에게 일을 나누는 방식이다.

잘 맞는 경우:

- 요청 종류가 다양하다.
- 어떤 Agent가 필요한지 매번 달라진다.
- 전체 진행 상태를 한곳에서 봐야 한다.

주의할 점:

- Supervisor가 너무 많은 일을 직접 하게 만들지 않는다.
- 배정 기준과 완료 기준을 분명히 해야 한다.

## 6. Hierarchical Delegation

큰 일을 작은 일로 쪼개고, 각 작은 일을 다시 담당자에게 맡기는 방식이다.

잘 맞는 경우:

- 작업 규모가 크다.
- 하위 작업 사이의 의존성이 많다.
- 여러 단계의 의사결정이 필요하다.

주의할 점:

- 처음 실습할 때는 너무 무거울 수 있다.
- 하위 작업의 완료 정의가 없으면 관리가 어려워진다.

## 동적 제어 패턴: 반복·합의·자기비판

순차·병렬·감독 같은 구조 패턴 외에, 결과 품질을 끌어올리는 "동적 제어" 패턴이 있다. 위험도가 높거나 합격 기준이 분명한 작업에서 골라 쓴다.

### 반복 검증 루프 (Evaluator-Optimizer)

Producer-Reviewer를 "한 번 검토"가 아니라 통과까지 반복하는 형태다. 생성 → 평가 → 미달이면 수정 → 다시 평가. 명확한 합격 기준이 있을 때 효과가 크다. 다만 무한 반복은 토큰 폭주, 에러 누적, 자기교정 역행(고칠수록 나빠짐)을 부르므로 반드시 종료 계약을 둔다.

| 출구 | 조건 | 처리 |
| --- | --- | --- |
| 통과 | 평가자가 합격 기준 충족 판정 | 정상 종료 |
| 횟수 상한 | 정한 N회 도달 (기본 2~3회) | 자동 통과 금지 → `미검증` 또는 `사람 승인 필요`로 에스컬레이션 |
| 수렴 정체 | 직전 라운드 대비 개선이 임계값 미만 | 더 못 올린다고 보고 멈춤 |
| 예산 상한 | 토큰·시간 한도 초과 | 현재 best와 사유를 보고하고 멈춤 |

추가 가드:

- best 보존과 롤백: 각 라운드 결과를 보존하고 "지금까지 최고"를 따로 둔다. 어떤 라운드가 점수를 떨어뜨리면 그 수정을 버리고 best로 되돌린다. 단조 개선을 가정하지 않는다.
- 외부 신호 결합: 매 라운드 평가를 테스트·체크리스트·결함 주입 회귀 세트에 묶어 자기평가 편향이 라운드마다 증폭되지 않게 한다.
- 두께 연동: 저위험은 루프를 생략하고 단발 검증, 고위험만 루프 + 미통과 시 사람 승인.

### 투표와 자기일관성 (Voting / Self-Consistency)

같은 작업을 여러 번 독립 실행해 다수결이나 best-of-N으로 고른다. 서로 다른 하위작업을 나누는 Fan-out/Fan-in과 다르다. 여기서는 같은 작업을 N회 돌린다. 분류·판정·환각 저감처럼 신뢰도가 중요한 단계에서 쓴다. 비용이 N배이므로 핵심 판정에만 제한적으로 쓴다.

### Reflection (자기비판)

별도 검토 Agent를 띄우지 않고, 한 Agent가 자기 출력을 비판하고 고치는 가벼운 루프다. Producer-Reviewer보다 싸지만, 외부 신호 없는 순수 자기비판은 한계가 있으므로 위험도가 낮거나 외부 기준(테스트·체크리스트)을 함께 줄 수 있을 때 쓴다.

### Routing (입력 분류 후 전문가 선택)

Expert Pool을 쓸 때 "어떤 전문가를 부를지"를 즉흥 판단하지 말고, 입력을 먼저 분류하는 단계를 1급으로 둔다. 분류 기준과 어디에도 안 맞을 때의 기본 경로를 명시하면 라우팅이 안정된다.

## 빠른 선택 기준

| 상황 | 추천 패턴 |
| --- | --- |
| 정해진 순서대로 처리한다 | Pipeline |
| 여러 자료를 나눠 읽고 합친다 | Fan-out/Fan-in |
| 필요할 때만 전문가가 필요하다 | Expert Pool |
| 초안과 검토를 분리해야 한다 | Producer-Reviewer |
| 요청마다 담당자가 달라진다 | Supervisor |
| 큰 프로젝트를 여러 하위 프로젝트로 쪼갠다 | Hierarchical Delegation |
