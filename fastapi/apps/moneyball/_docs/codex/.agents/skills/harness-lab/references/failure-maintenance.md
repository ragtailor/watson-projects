# 실패와 유지보수 가이드

이 문서는 기존 Codex 하네스를 점검하거나, 만든 하네스가 잘 작동하지 않을 때 사용한다. 목표는 "AI가 못했다"에서 멈추지 않고, 어느 하네스 층을 고쳐야 하는지 찾는 것이다.

## 목차

1. 언제 읽을까
2. 하네스 도입 실패 10가지
3. 실패했을 때 다시 세우는 순서
4. 기존 하네스 감사 절차
5. Drift 점검표
6. 변경 이력 형식
7. 개선 기록 형식
8. ADR 형식 (큰 구조 결정)
9. 유지보수 판단 기준
10. 하네스 덜어내기 진단
11. 운영 원칙

## 언제 읽을까

- 사용자가 "기존 하네스를 점검해줘", "왜 잘 안 돌아가는지 봐줘"라고 할 때
- `AGENTS.md`, `.agents/skills`, `.codex/agents` 사이의 불일치를 확인할 때
- 실패 사례를 바탕으로 Agent, Skill, Orchestrator를 개선할 때
- 변경 이력과 개선 기록을 남겨야 할 때

## 하네스 도입 실패 10가지

| 번호 | 실패 | 증상 | 고칠 곳 |
| --- | --- | --- | --- |
| 1 | 프롬프트만 계속 고친다 | 좋은 결과가 반복되지 않는다 | 목표, 자료, 검증 기준을 Skill/Orchestrator로 꺼낸다 |
| 2 | 한 번에 너무 크게 만든다 | 구조는 크지만 실제로 쓰지 못한다 | 반복 업무 하나로 범위를 줄인다 |
| 3 | 자료 없이 판단을 맡긴다 | 그럴듯하지만 근거가 약하다 | 컨텍스트 수집 카드와 근거 표시를 추가한다 |
| 4 | 중간 산출물이 없다 | 어디서 틀렸는지 모른다 | 요약표, 매트릭스, 초안, 검토표를 둔다 |
| 5 | 검증 기준이 머릿속에만 있다 | 평가가 매번 흔들린다 | 체크리스트와 테스트 프롬프트를 만든다 |
| 6 | 실패 흐름이 없다 | 오류가 나면 처음부터 다시 한다 | 재시도, 질문, 중단, 사람 인계를 정한다 |
| 7 | 사람 승인 지점이 없다 | 외부 발송·제출이 불안하다 | 승인 조건을 Orchestrator에 넣는다 |
| 8 | 도구와 권한이 너무 넓다 | 필요 없는 파일과 기능까지 건드린다 | Agent별 최소 도구와 금지 행동을 둔다 |
| 9 | 메모리와 기록을 방치한다 | 오래된 규칙이 계속 영향을 준다 | 변경 이력과 정리 루프를 둔다 |
| 10 | 실제 업무 입구에 붙지 않는다 | 문서는 있지만 아무도 쓰지 않는다 | `AGENTS.md` 자연어 라우팅과 사용 예시를 보강한다 |

## 실패했을 때 다시 세우는 순서

1. 하네스가 맡은 일을 한 문장으로 다시 줄인다.
2. 실제로 자주 쓰는 입력 자료만 남긴다.
3. 최종 결과 전에 필요한 중간 산출물 하나를 추가한다.
4. 검증 기준을 3-5개로 다시 쓴다.
5. 사람 승인 지점을 하나만 분명히 정한다.
6. 한 번 실행해 보고 실패 기록을 남긴다.
7. 다음 버전에 반영할 것 하나만 고친다.

한 번에 모든 것을 고치지 않는다. 가장 큰 실패 하나를 잡고, 해당 Agent, Skill, Orchestrator 중 한 곳만 먼저 수정한다.

## 기존 하네스 감사 절차

기존 하네스를 개선할 때는 먼저 읽고 나서 수정한다.

1. `AGENTS.md`에서 하네스 포인터와 자연어 라우팅 규칙을 확인한다.
2. `AGENTS.override.md`가 있으면 로컬 임시 지시가 현재 하네스와 충돌하지 않는지 확인한다.
3. `.agents/skills/` 목록과 Orchestrator Skill을 확인한다.
4. `.codex/agents/` 목록을 확인한다.
5. Orchestrator의 Agent 목록과 실제 Agent TOML 파일이 일치하는지 본다.
6. Orchestrator의 Skill 목록과 실제 Skill 폴더가 일치하는지 본다.
7. 각 Skill의 frontmatter `name`, `description`이 폴더명과 충돌하지 않는지 본다.
8. 테스트 프롬프트, 실패 처리, 사람 승인 조건이 있는지 확인한다.
9. 오래된 Agent, 쓰이지 않는 Skill, 중복된 규칙을 표시한다.

## Drift 점검표

| 점검 항목 | 좋은 상태 | 위험 신호 |
| --- | --- | --- |
| `AGENTS.md` 포인터 | 하네스 이름과 Orchestrator가 맞다 | 예전 Skill 이름을 가리킨다 |
| `AGENTS.override.md` | 필요한 로컬 임시 지시만 있다 | 프로젝트 공통 규칙을 몰래 덮어쓴다 |
| Agent 목록 | Orchestrator가 부르는 Agent가 실제 존재한다 | 삭제된 Agent를 아직 호출한다 |
| Skill 목록 | 작업 Skill과 Orchestrator 책임이 구분된다 | Orchestrator가 일반 작업까지 모두 직접 한다 |
| Description | 자연어 요청과 제외 조건이 있다 | 너무 짧거나 너무 넓다 |
| 산출물 경로 | 단계별 파일명이 정해져 있다 | 중간 결과가 대화에만 남는다 |
| 산출물 저장 | 파일을 맡은 Agent가 실제로 그 파일을 저장한다 | 저장한 척 본문 요약으로 대체한다 |
| 검증 | 테스트와 체크리스트가 있다 | "검토한다"만 있고 기준이 없다 |
| 승인 | 위험 행동 전에 멈춘다 | 외부 발송, 제출, 삭제를 자동으로 끝낸다 |
| 기록 | 변경 이력이 있다 | 언제 왜 바꿨는지 모른다 |

## 변경 이력 형식

`AGENTS.md` 또는 하네스 문서의 짧은 섹션에 아래 표를 둔다. 기존 프로젝트 규칙이 있으면 그 형식을 우선한다.

```md
## 하네스 변경 이력

| 날짜 | 변경 내용 | 대상 | 사유 |
| --- | --- | --- | --- |
| 2026-05-13 | 초기 구성 | 전체 | 반복 업무 하네스 생성 |
| 2026-05-13 | 검토 Agent 추가 | `.codex/agents/reviewer.toml` | 제출 전 누락 확인 필요 |
```

변경 이력은 길게 쓰는 일지가 아니다. 다음 세션에서 "왜 이 파일이 생겼는지" 알 수 있을 정도면 충분하다.

## 개선 기록 형식

실행 후 피드백을 다음 버전 재료로 남긴다.

```md
## 개선 기록

- 날짜:
- 실행한 요청:
- 기대한 결과:
- 실제 결과:
- 잘된 점:
- 막힌 점:
- 다음 버전에서 바꿀 규칙:
```

좋은 기록은 다음 버전에서 바꿀 규칙까지 연결된다.

```md
- Reviewer가 대상 독자 수준을 확인하지 않았다.
- 다음 버전에서는 Reviewer Skill에 "독자가 모를 수 있는 용어를 표시하고 쉬운 말로 바꾸기"를 추가한다.
```

## ADR 형식 (큰 구조 결정)

큰 구조 결정(Agent Team vs 단일 흐름, 하네스 두께, 패턴 선택, 역할 분리·병합, 검증 조각 추가·제거)은 변경 이력 한 줄로 끝내지 말고 ADR로 남긴다. 파일은 `artifacts/decisions/ADR-{번호}-{제목}.md`에 두거나 개선 기록 문서 안에 넣는다.

```md
# ADR-001: {결정 제목}

- 날짜: YYYY-MM-DD
- 상태: 제안 | 채택 | 폐기 | 대체됨(→ ADR-00N)

## 맥락
- 어떤 문제·제약 때문에 결정이 필요했는가.

## 결정
- 무엇으로 정했는가.

## 버린 대안과 이유
- 대안 A — 버린 이유.
- 대안 B — 버린 이유.

## 가정·트레이드오프
- 이 결정이 성립하는 전제와, 받아들인 비용.

## 재검토 조건
- 어떤 신호(모델 향상, 업무 변화, 품질 저하)가 보이면 이 결정을 다시 연다.
```

변경 이력은 "무엇이 언제 바뀌었나"(시간 순 추적)이고, ADR은 "왜 이렇고 무엇을 포기했나"(결정의 근거)이다. 역할이 다르므로 함께 쓴다. 특히 **재검토 조건**은 하네스 덜어내기 진단에서 각 조각의 제거 여부를 판단하는 직접 근거가 된다.

## 유지보수 판단 기준

| 피드백 유형 | 수정 대상 | 예시 |
| --- | --- | --- |
| 결과물 품질이 낮다 | 해당 작업 Skill | 분석이 얕다 -> 근거와 비교 기준 추가 |
| 역할이 빠졌다 | Agent 정의 또는 새 Agent | 보안 검토가 필요하다 -> reviewer 추가 |
| 순서가 어색하다 | Orchestrator Skill | 검증을 초안 뒤가 아니라 자료 정리 뒤로 이동 |
| 스킬이 안 켜진다 | Skill description | 사용자가 쓰는 표현을 트리거에 추가 |
| 엉뚱하게 켜진다 | description 제외 조건 | "단순 맞춤법 교정에는 사용하지 않음" 추가 |
| 위험 행동이 불안하다 | 승인 조건과 도구 범위 | 외부 발송 전 사람 승인 추가 |
| 같은 실패가 반복된다 | 테스트와 개선 기록 | 실패 프롬프트를 회귀 테스트로 저장 |

## 하네스 덜어내기 진단

하네스는 두꺼워지기만 하면 안 된다. 모든 Agent, Skill, 검증 단계는 "모델이 혼자서는 잘 못 하는 것"을 대신 채우는 장치다. 모델이 좋아지거나 업무가 단순해지면, 한때 필요했던 조각이 이제는 토큰만 쓰고 품질에는 기여하지 않을 수 있다. 점검할 때는 추가뿐 아니라 제거 후보도 함께 찾는다.

| 조각 | 남길 신호 | 덜어낼 신호 |
| --- | --- | --- |
| 특정 Agent | 빠지면 품질·안전이 눈에 띄게 떨어진다 | 다른 Agent나 Orchestrator 단계로 흡수해도 결과가 같다 |
| 특정 Skill | 반복 절차가 분명하고 여러 곳에서 쓴다 | 한 번만 쓰이고 본문이 일반 지시와 다를 바 없다 |
| 특정 검증 단계 | with/without에서 실제 오류를 잡아낸다 | 양쪽에서 항상 통과해 차별력이 없다 |
| 두꺼운 지시·제약 | 없으면 결과가 흔들린다 | 모델이 이미 일관되게 지키는 내용을 반복 지시한다 |

제거는 추측으로 하지 않는다. 의심되는 조각을 하나만 빼고 대표 프롬프트로 with/without 비교를 돌려 품질 영향을 확인한 뒤, 영향이 없으면 제거 후보로 표시한다. 한 번에 여러 조각을 빼면 어느 것이 영향을 줬는지 알 수 없으므로 하나씩 검증한다. 제거한 조각과 사유는 개선 기록과 변경 이력에 남겨, 나중에 되살릴 근거로 둔다.

## 운영 원칙

- 문서가 실제 사용 흐름에 붙어 있어야 한다. 쓰지 않는 하네스는 좋은 하네스가 아니다.
- 오래된 규칙은 품질을 떨어뜨릴 수 있다. 필요 없는 Agent와 Skill은 정리 후보로 표시한다.
- 하네스는 계속 두꺼워지기만 하면 안 된다. 모델이나 업무가 좋아져 더 단순해도 되는 부분은 줄인다.
- 위험한 작업일수록 검증, 승인, 기록을 두껍게 둔다.
- 작은 실패를 부끄럽게 보지 않는다. 실패 기록은 다음 하네스를 더 좋게 만드는 재료다.
