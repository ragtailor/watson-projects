## 기존 확장 시 다시 볼 Phase

신규 구축이 아니라 기존 하네스에 무언가를 추가·수정하는 경우에는 Phase 0-7을 처음부터 다 돌리지 않는다. 변경 유형별로 필요한 Phase만 다시 본다.

| 변경 유형 | 다시 볼 Phase |
| --- | --- |
| Agent 추가 | Phase 3, Phase 4, Phase 6, Phase 7 |
| Skill 추가/수정 | Phase 5, Phase 6, Phase 7 |
| Orchestrator 흐름 변경 | Phase 3, Phase 6, Phase 7 |
| 산출물 구조 변경 | Phase 2, Phase 6, Phase 7 |
| 검증 강화 | Phase 4, Phase 6, Phase 7 |
| 큰 아키텍처 변경 | Phase 1부터 다시 청사진을 만든다 |

기존 확장도 바로 파일을 고치지 않는다. 먼저 현재 구조와 바꿀 부분을 요약하고, 개선 청사진을 보여준 뒤 사용자가 적용을 요청하면 수정한다.

## Phase 0. 현재 상황 확인

확인할 것:

- 사용자가 원하는 결과가 설명, 청사진, 실제 파일 생성 중 무엇인지 확인한다. 다만 하네스 생성 요청은 실제 파일 생성을 원하더라도 청사진을 먼저 보여준 뒤 승인받는다.
- 사용자의 하네스 경험 수준과 Codex 사용 경험을 파악한다.
- 현재 프로젝트에 `AGENTS.md`, `AGENTS.override.md`, `.agents/skills`, `.codex/agents`가 있는지 확인한다.
- `.codex/config.toml`은 선택 설정이므로, 이미 있거나 사용자가 실행 설정까지 원할 때만 확인한다.
- 기존 파일이 있으면 읽고, 덮어쓰기 전에 변경 범위를 설명한다.
- 사용자의 일이 하네스 성숙도 1-5단계 중 어디에 가까운지 가볍게 진단한다.
- 사용자가 이미 설계 카드나 실습 키트 내용을 제공했다면 Phase 2의 입력으로 사용한다.

좋은 질문:

- "이 하네스는 어떤 일을 돕기 위한 것인가요?"
- "결과물이 글, 표, 코드, 계획서, 검토 보고서 중 무엇에 가까운가요?"
- "처음 실습이라면 아주 작은 업무 하나로 시작해도 괜찮습니다."
- "AI가 지금까지 이 일을 할 때 자주 흔들린 지점은 무엇인가요?"
- "사람이 반드시 확인해야 하는 지점이 있나요?"

성숙도와 설계 카드가 필요하면 `harness-design-cards.md`를 읽는다.

## 청사진 승인 게이트

Phase 0-7은 먼저 청사진으로 제시한다. 청사진 단계에서는 파일을 만들지 않는다.

구성 단계로 넘어가는 조건:

- 직전 응답에서 청사진을 제시했다.
- 사용자가 그 청사진에 대해 "좋아", "이 구조로 만들어줘", "승인해", "실제로 구성해줘"처럼 명시적으로 승인했다.

구성 단계로 넘어가면 안 되는 경우:

- 청사진을 아직 보여주지 않았다.
- 사용자가 첫 요청에서 "바로 만들어줘", "분석 없이 생성해", "파일로 만들어줘"라고만 말했다.
- 사용자가 의견이나 방향만 물었다.

이 경우에는 청사진을 먼저 만들고, 마지막에 구성 여부를 묻는다.

## Phase 1. 도메인 분석

도메인 분석은 "이 일이 어떤 일인가"를 읽는 단계다. 역할 분리, 사람 승인 지점, 적합성 판정이 모두 여기서 나오므로 하네스 설계에서 가장 중요한 분석이다.

먼저 사용자의 일을 사람의 업무로 풀어쓴다. 예를 들어 "블로그 글 작성"은 주제 정함 → 독자 정함 → 자료 모음 → 목차 → 초안 → 검토로 나눈다. 이 일상 언어 풀이 위에서 아래를 함께 읽는다.

| 분석 항목 | 무엇을 보나 | 무엇을 결정하나 |
| --- | --- | --- |
| 작업 유형 | 생성·검증·편집·분석·조사 중 무엇인가 | 팀 패턴 선택 |
| 기존 자료·코드베이스 | 코드면 기술 스택·데이터 모델·주요 모듈, 문서 업무면 기존 양식·자료·입력 형식·제약 | 컨텍스트, 재사용 가능 범위 |
| 반복 보조 작업 | 매번 반복되는 조사·정리·검토 | 역할(Agent) 후보 |
| 위험 지점·외부 의존 | 외부 발송·제출·삭제·결제·개인정보·법적 표현 | 사람 승인 지점, 하네스 두께 |
| 기존 하네스 충돌 | 기존 Agent·Skill과 겹침, drift | 신규/확장/운영 분기 |
| 숙련도·맥락 단서 | 사용자가 쓰는 용어, 질문 수준 | 설명 밀도 |

이 단계에서는 Agent나 Skill이라는 말을 너무 빨리 꺼내지 않는다. 먼저 사람의 절차와 위 항목을 정리한 뒤 다음 단계의 입력으로 넘긴다. 특히 도메인 분석은 적합성 게이트의 입력이다 — 반복성·구조 안정성·검증 가능성·외부 신호를 여기서 읽어 "이 업무가 하네스화할 가치가 있는가"를 판정한다. 코드 프로젝트뿐 아니라 일상 문서·운영 업무에도 똑같이 적용한다.

## Phase 2. 산출물 정의

하네스는 결과물이 분명할수록 좋아진다. 이 단계에서는 최종 결과만 정하지 말고 하네스 7요소를 함께 확인한다.

정의할 것:

- 최종 산출물: 글, 보고서, 계획표, 코드 변경, 체크리스트 등
- 중간 산출물: 조사 노트, 구조안, 초안, 검토표 등
- 산출물 계약: 각 산출물의 파일 경로, 만드는 역할, 다음에 읽는 역할, 재사용 방법
- 성공 기준: 정확성, 이해 쉬움, 누락 없음, 실행 가능성 등
- 실패 기준: 근거 없음, 너무 장황함, 독자 수준 불일치 등
- 컨텍스트: AI가 먼저 봐야 할 자료와 사용자가 나중에 제공해도 되는 자료
- 도구와 권한: 읽기, 쓰기, 실행, 외부 발송, 삭제 중 무엇이 필요한지
- 사람 승인 지점: 게시, 제출, 발송, 결제, 삭제, 법적 표현, 개인정보 처리 등
- 기록과 개선: 다음번에 재사용할 예시, 실패 기록, 변경 이력

하네스 7요소 표가 필요하면 `harness-design-cards.md`를 읽는다.

산출물 계약은 아래처럼 간단히 잡는다.

| 단계 | 파일 | 만드는 역할 | 다음에 읽는 역할 | 재사용 방식 |
| --- | --- | --- | --- | --- |
| 입력 정리 | `artifacts/00-input.md` | Orchestrator | 모든 역할 | 요청 범위와 제약 확인 |
| 조사/분석 | `artifacts/01-brief.md` | Researcher 또는 Analyst | Writer 또는 Builder | 초안 작성의 근거 |
| 초안 | `artifacts/02-draft.md` | Writer 또는 Builder | Reviewer | 검토 대상 |
| 검토 | `artifacts/03-review.md` | Reviewer | Orchestrator | 수정·승인 판단 |
| 최종 | `artifacts/final.md` | Orchestrator | 사용자와 다음 실행 | 실제 사용 결과 |

이름은 도메인에 맞게 바꿀 수 있지만, 실행 하네스라면 "누가 무엇을 파일로 남기고 다음 단계가 무엇을 읽는지"는 반드시 있어야 한다.

## Phase 3. 팀 패턴 선택

업무 성격에 따라 패턴을 고른다. 먼저 단일 흐름으로 가능한지 확인하고, 분업 이유가 분명할 때만 여러 Agent를 둔다.

- 순서가 중요한 일: Pipeline
- 여러 자료를 나눠 봐야 하는 일: Fan-out/Fan-in
- 여러 전문 관점이 필요한 일: Expert Pool
- 만들고 검토해야 하는 일: Producer-Reviewer
- 요청마다 담당자를 골라야 하는 일: Supervisor
- 큰 일을 여러 하위 작업으로 쪼개야 하는 일: Hierarchical Delegation

자세한 기준은 `pattern-catalog.md`를 읽는다.

커스텀 Agent 분리가 필요한지 애매하면 `agent-skill-design.md`의 문맥 경계 기준을 읽는다.

## Phase 4. Agent 설계

Agent는 "누가 이 일을 맡는가"를 정하는 문서다. Codex에서는 역할이 분명한 팀원을 `.codex/agents/{agent-name}.toml`로 만들 수 있다.

각 Agent에 포함할 것:

- 역할 이름
- 맡는 일
- 입력으로 받는 것
- 출력으로 남기는 것
- 잘못하면 안 되는 것
- 필요한 도구나 파일 접근 범위

첫 실습에서는 3-4개 Agent가 적당하다.

Agent를 꼭 파일로 만들 필요가 없는 경우도 있다. 한 번 설명만 해보는 청사진이면 역할표만으로 충분하다. 반복해서 같은 역할을 부르고 싶거나, 특정 역할의 말투와 판단 기준을 고정하고 싶을 때 `.codex/agents` 파일로 만든다.

Agent를 별도로 만들기 전에는 아래를 확인한다.

- 입력과 출력이 독립적인가?
- 다른 전문성이 필요한가?
- 반복해서 다시 쓰일 책임인가?
- 빠지면 품질이나 안전에 큰 문제가 생기는가?
- 같은 정보를 계속 이어서 봐야 하는 일은 아닌가?

자세한 판단 기준과 Agent TOML 구조는 `agent-skill-design.md`를 읽는다.

## Phase 5. Skill 설계

Skill은 "그 역할이 어떻게 일하는가"를 정하는 매뉴얼이다. Codex에서는 실행 스킬을 `.agents/skills/{skill-name}/SKILL.md`로 만든다.

각 Skill에 포함할 것:

- 언제 이 Skill을 써야 하는지
- 작업 순서
- 확인해야 할 기준
- 산출물 형식
- 애매할 때 멈추고 물어볼 조건
- 쓰면 안 되는 요청 또는 역할 밖 행동
- 정상 입력과 예외 입력 예시

처음 실습할 때는 긴 매뉴얼보다 짧은 체크리스트가 더 효과적이다.

Codex Skill의 `description`은 매우 중요하다. Codex가 스킬을 고를 때 먼저 보는 부분이기 때문이다. "언제 이 스킬을 써야 하는지", "비슷하지만 쓰면 안 되는 경우는 무엇인지"를 짧고 분명하게 적는다.

Skill description, 출력 형식, 부정 케이스 설계가 필요하면 `agent-skill-design.md`를 읽는다.

## Phase 6. Orchestrator 설계

Orchestrator는 여러 역할의 순서와 전달물을 관리한다.

Orchestrator 역할을 하는 Skill은 폴더명과 frontmatter `name`을 `{하네스-이름}-orchestrator` 형식으로 만든다. 예를 들어 가족 여행 준비 하네스라면 `.agents/skills/trip-planning-orchestrator/SKILL.md`처럼 둔다. 일반 작업 Skill에는 `-orchestrator` 접미사를 붙이지 않는다.

생성되는 `AGENTS.md`에는 자연어 요청을 이 Orchestrator Skill로 먼저 연결하는 규칙을 함께 남긴다. 사용자가 `$trip-planning-orchestrator` 같은 이름을 외우지 않아도 "가족 여행 일정 다시 정리해줘"처럼 말하면 전체 진행표를 먼저 타도록 하기 위해서다.

반드시 정할 것:

- 시작 입력
- 단계별 담당 Agent
- 단계별 산출물 파일 이름
- `artifacts/README.md` 또는 같은 역할의 산출물 지도가 어떤 파일을 설명하는지
- 다음 단계로 넘기는 정보
- 실패했을 때 다시 실행할 조건
- 사람이 확인해야 하는 지점
- 기존 산출물이 있을 때 초기 실행, 부분 재실행, 새 실행 중 어떤 모드인지 판단하는 기준
- `AGENTS.md`에는 전체 세부 규칙이 아니라 Orchestrator로 이어지는 포인터를 남기는 방식

Orchestrator는 모든 일을 직접 하는 존재가 아니다. 일을 나누고, 이어주고, 빠진 부분을 확인하는 팀장 역할이다.

Codex에서는 Orchestrator도 하나의 Skill로 만드는 것이 가장 단순하다. 사용자는 자연어로 요청해도 되고, 필요하면 `$travel-planning-orchestrator`처럼 입구 스킬을 직접 호출할 수도 있다. 어느 경우든 Orchestrator가 필요한 역할과 산출물을 안내하게 한다.

실행 하네스에서는 Orchestrator가 먼저 `artifacts/`를 확인한다. 기존 산출물이 있으면 이어서 고칠지, 일부 단계만 다시 돌릴지, 새 실행으로 보존할지 정한다. 대화에만 남은 결과는 다음 실행이 읽을 수 없으므로 산출물로 보지 않는다.

### 드라이런: 설계 검증

파일을 만들기 전에 흐름 그래프를 점검한다. (a) Phase 순서가 논리적인가, (b) 각 단계의 입력이 이전 단계의 출력과 실제로 맞물리는가(끊긴 dead-link 없는가), (c) 에러·미승인 시 폴백 경로가 실행 가능한가. 끊긴 구간이 있으면 산출물 계약을 고친 뒤 진행한다.

### 긴 작업: 컨텍스트 리셋과 핸드오프

단계가 길게 이어지는 하네스에서는 한 대화에 모든 맥락을 쌓다가 컨텍스트가 가득 차는 문제가 생긴다. 컨텍스트가 길어지면 모델은 작업을 조기에 끝내려 하거나 앞쪽 정보를 흐리게 다룬다. 요약으로 계속 이어붙이는 것은 연속성은 주지만 깨끗한 시작은 주지 못한다.

긴 작업은 Phase 단위로 끊고, 단계 사이에 핸드오프 산출물(`artifacts/handoff.md` 등)을 남긴 뒤 깨끗한 컨텍스트로 다시 시작한다. 핸드오프 산출물에는 지금까지의 결정과 근거, 다음 단계가 읽을 입력 파일 경로, 남은 할 일과 미검증 영역, 사람 승인이 걸린 지점과 현재 승인 상태를 담는다. 그러면 다음 단계는 전체 대화를 다시 읽지 않고 핸드오프 산출물과 필요한 산출물만 읽어 이어받는다. 단계마다 착수 전에 "이 단계가 끝나면 무엇이 있어야 하는가"를 성공 계약으로 먼저 정해, 리셋 후에도 기준이 흔들리지 않게 한다.

## Phase 7. 테스트와 개선

테스트는 최소 3개를 만든다.

- 정상 사례: 잘 정의된 요청
- 애매한 사례: 범위가 흐린 요청
- 실패하기 쉬운 사례: 자료가 부족하거나 조건이 충돌하는 요청

각 테스트에는 기대 결과와 확인 기준을 붙인다.

실무용 하네스라면 아래도 추가한다.

- 부정 테스트: 이 Skill이나 Orchestrator가 실행되면 안 되는 요청
- 반복 테스트: 같은 입력을 여러 번 넣어도 구조가 안정적인지 확인
- 트리거 테스트: 사용자가 실제로 말할 법한 표현에서 Skill이 잘 선택되는지 확인

사용 후에는 실패나 불편을 개선 기록으로 남긴다. 기존 하네스를 고쳤다면 변경 이력도 남긴다.

피드백을 받으면 유형별로 어디를 고칠지 정한다: 결과물 품질 → 해당 Skill, 빠진/겹친 역할 → Agent 또는 Orchestrator 구성, 어색한 순서 → Orchestrator, 안 켜짐/엉뚱하게 켜짐 → description, 반복 실패 → 테스트·회귀 세트. 상세 표는 `failure-maintenance.md`의 유지보수 판단 기준을 읽는다.

진화는 사용자가 "고쳐줘"라고 말할 때만이 아니라, 아래 신호를 보면 먼저 제안한다 — (a) 같은 유형의 피드백이 2회 이상 반복, (b) 특정 Agent·단계가 반복적으로 실패, (c) 사용자가 Orchestrator를 우회해 수동으로 처리하는 것이 관찰됨.

Codex용 하네스에서는 아래도 함께 확인한다.

- 새 Orchestrator Skill을 `$skill-name`으로 부를 수 있는가?
- `AGENTS.md`가 새 하네스 위치를 정확히 가리키는가?
- `.codex/agents`의 역할이 서로 겹치지 않는가?
- 실행 산출물이 `artifacts/`처럼 다음 단계가 이어받기 쉬운 위치에 남는가?
- `artifacts/README.md`, 단계별 중간 산출물, 최종 산출물이 생겼는가?
- 실행 하네스 파일이 `AGENTS.md`, `.agents/skills`, `.codex/agents`, `artifacts/` 기준으로 정리되었는가?

기존 하네스 점검, drift, 변경 이력 기준이 필요하면 `failure-maintenance.md`를 읽는다.
