## 결함 주입과 회귀 세트

검토 단계가 매번 통과만 한다면 두 가지 원인이 섞여 있다. (1) 모델이 좋아져 앞 단계 출력이 실제로 깨끗해진 진짜 통과거나, (2) 검토 기준이 무뎌 실제 결함도 통과시키는 거수기다. 이 둘을 가르지 않고 검토를 빼면 사고가 난다. 가르는 가장 빠른 방법이 **결함 주입(fault injection)**이다.

결함 주입은 일부러 망가뜨린 입력을 흘려보내 검증 단계가 그 결함을 잡는지 보는 것이다. 잡으면 살아 있는 검토, 그냥 통과시키면 죽은 검토다. 이 한 가지 테스트가 "모델이 좋아져서 통과"와 "검토가 망가져서 통과"를 가장 깔끔하게 가른다. `failure-maintenance.md`의 하네스 덜어내기 진단에서 always-pass 검증을 식별할 때 함께 쓴다.

산출물 종류별 결함 예시:

| 산출물 | 심을 결함 | 살아 있는 검토라면 |
| --- | --- | --- |
| 문서·콘텐츠 | 사실 오류, 깨진 인용·링크, 톤 이탈, 금지 표현, 누락 섹션 | 해당 항목을 위치·근거와 함께 탈락시킨다 |
| 수치·보고서 | 합계 불일치, 단위 혼용, 직전 분기 대비 근거 누락 | 불일치를 잡아낸다 |
| 코드 | 응답 shape 변조, 깨진 라우트, 누락 상태 전이 | 생산자-소비자 계약 불일치를 잡는다 |
| 제출·발송물 | 잘못된 수신자, 빠진 첨부, 미승인 상태 | 승인 게이트에서 멈춘다 |

결함 주입 입력은 한 번 쓰고 버리지 않는다. **회귀 세트**로 `artifacts/evals/regression/`에 보관하고, 프롬프트·모델·하네스 구조를 바꿀 때마다 다시 흘려보내 검토가 여전히 잡는지 확인한다. 이것이 모델이 바뀌어도 하네스가 늙지 않게 하는 핵심 장치다.

검토 프롬프트는 "문제없으면 PASS"가 아니라 "막을 이유를 찾아라(반려 우선)"로 뒤집는다. 기본값을 반려로 두고, 발견 항목 목록·위치·근거·심각도를 강제로 출력하게 하면 빈 결과가 진짜인지 게으름인지 드러난다. 분량·섹션 존재·링크 유효성·금지어처럼 객관적인 항목은 LLM 검토 대신 결정적 체크(체크리스트·스크립트)로 빼고, LLM 검토는 논리·톤·사실성 같은 판단 영역에 집중시킨다.

## With/Without 하네스 A/B 비교

하네스가 정말 가치를 더하는지 보려면, 같은 요청을 하네스로 한 번, 하네스 없이 한 번 실행해 나란히 비교한다. "좋아 보인다"가 아니라 "하네스를 썼을 때 무엇이 더 나아졌는가"를 증거로 확인한다.

### 실행 구조

각 테스트 프롬프트마다 서브에이전트 두 개를 같은 입력으로 동시에 띄운다.

| 구성 | 입력 | 사용 하네스 | 출력 경로 |
| --- | --- | --- | --- |
| With-harness | {동일 프롬프트} | Orchestrator·Skill 사용 | `artifacts/evals/iteration-N/{eval-name}/with-harness/` |
| Baseline | {동일 프롬프트} | 하네스 없이 | `artifacts/evals/iteration-N/{eval-name}/baseline/` |

두 실행의 프롬프트와 입력 자료는 완전히 같아야 한다. 입력이 달라지면, 결과 차이가 하네스 덕분인지 입력 차이 때문인지 구분할 수 없다.

### Baseline 고르기

| 상황 | Baseline |
| --- | --- |
| 새 하네스를 처음 만들 때 | 하네스 없이 같은 프롬프트 실행 |
| 기존 하네스를 개선할 때 | 수정 전 버전(Agent·Skill 스냅샷 보존) |

### 무엇을 기록할까

- 산출물은 두 경로에 각각 저장한다.
- 완료 알림의 `total_tokens`, `duration_ms`를 그 자리에서 저장한다. 알림 시점이 지나면 복구할 수 없다.
- with-harness가 baseline보다 나은지, 같은지, 오히려 나쁜지 판정한다.

비교 기준:

- 산출물이 더 구조적인가?
- 빠진 정보가 줄었는가?
- 검토 단계가 실제 오류를 잡았는가?
- 사용자가 다음 행동을 하기 쉬워졌는가?
- 설명이 독자 수준에 맞는가?
- 토큰과 시간이 더 들었다면, 그만큼의 품질 향상이 있었는가?

양쪽 구성에서 항상 통과하는 검증 항목은 하네스의 차별적 가치를 못 본다. 그런 항목은 하네스가 있어야만 통과하는 더 도전적인 항목으로 바꾼다.

## 체크리스트

```md
## 하네스 테스트 체크리스트

- [ ] 요청의 목적을 다시 말해준다.
- [ ] 최종 산출물 형식을 분명히 한다.
- [ ] 역할이 너무 많거나 적지 않다.
- [ ] Agent와 Skill의 책임이 겹치지 않는다.
- [ ] Orchestrator가 중간 산출물을 이어준다.
- [ ] `artifacts/README.md`에 산출물 지도와 다음 단계가 읽을 파일이 적혀 있다.
- [ ] 단계별 중간 산출물과 최종 산출물이 대화가 아니라 파일로 남는다.
- [ ] Orchestrator Skill 폴더명과 `name`이 `-orchestrator`로 끝난다.
- [ ] `AGENTS.md`에 자연어 요청을 Orchestrator Skill로 연결하는 규칙이 있다.
- [ ] 가능하면 같은 프롬프트로 with-harness와 baseline을 비교하고, 토큰·시간 대비 품질 향상을 근거로 남긴다.
- [ ] 트리거 검증에 경계가 애매한 near-miss should-not-trigger와 기존 Skill 충돌 확인이 포함된다.
- [ ] 애매한 요청에서 필요한 질문을 한다.
- [ ] 실패 위험 사례에서 추측을 단정하지 않는다.
- [ ] 항상 통과하는 검토 단계가 있으면 결함 주입으로 살아 있는지 확인하고, 결함 입력을 회귀 세트로 보관한다.
- [ ] 개선할 점을 기록한다.
- [ ] 외부 발송, 제출, 삭제, 결제, 개인정보 처리에는 사람 승인 지점이 있다.
- [ ] 승인 지점에서 실제로 사용자에게 승인을 요청하고 멈춘다. 라벨만 붙이거나 묻지 않고 통과하지 않는다(승인 게이트가 능동적으로 동작하는지 부정 테스트로 확인).
- [ ] Skill description에 써야 할 상황과 쓰면 안 되는 상황이 함께 있다.
- [ ] Agent와 Skill의 책임이 `누가`와 `어떻게`로 분리되어 있다.
- [ ] Orchestrator Skill을 `$skill-name`으로 호출하는 예시가 있다.
- [ ] `AGENTS.md`가 생성된 하네스 위치를 정확히 가리킨다.
- [ ] 실행 하네스 파일이 `AGENTS.md`, `.agents/skills`, `.codex/agents`, `artifacts/` 기준으로 정리되어 있다.
```

## 트리거 테스트 — near-miss 중심

Skill이 실행되지 않거나 엉뚱하게 실행되는 문제는 대부분 description에서 시작된다. 핵심은 명백한 정답이 아니라, 경계가 애매한 near-miss를 가려내는 것이다. 각 Skill마다 아래를 만든다.

| 테스트 | 개수 | 목적 |
| --- | ---: | --- |
| Should-trigger | 5-8개 | 사용자가 실제로 말할 법한 표현에서 Skill이 켜지는지 확인 |
| Should-not-trigger | 5-8개 | 비슷하지만 다른 업무에서 Skill이 끼어들지 않는지 확인 |

쿼리 작성 기준:

- 실제 사용자가 입력할 법한 구체적이고 자연스러운 문장으로 쓴다. 파일명, 회사명, 개인 맥락 같은 디테일을 넣는다.
- 톤(공식·캐주얼), 길이, 형식을 섞고, 명백한 정답보다 경계 케이스에 집중한다.

Should-not-trigger는 명백히 무관한 요청보다 경계가 애매한 near-miss가 좋다. "피보나치 함수 작성"처럼 무관한 문장은 테스트 가치가 없다. 예를 들어 제안서 Skill이라면 "내부 프로젝트 계획서 써줘"가, 산출물 하네스라면 "이 엑셀의 차트를 PNG로 추출만 해줘"가 좋은 near-miss다.

기존 Skill 충돌 검증: 새 하네스의 description이 기존 `.agents/skills/`의 트리거 영역과 겹치지 않는지 확인한다. should-trigger 쿼리가 기존 Skill을 잘못 켜면, description에 "쓰면 안 되는 상황"을 더 분명히 적어 경계를 만든다.

## 흔한 실패와 수정 방향

| 실패 | 원인 | 수정 |
| --- | --- | --- |
| 결과가 산만하다 | 최종 산출물 정의가 흐리다 | Phase 2에서 산출물 형식을 먼저 정한다 |
| Agent가 서로 같은 일을 한다 | 역할 경계가 겹친다 | Agent별 입력과 출력을 다시 나눈다 |
| 검토가 약하다 | 리뷰 기준이 없다 | Reviewer Skill에 체크리스트를 넣는다 |
| 실행 순서가 헷갈린다 | Orchestrator가 약하다 | 단계별 산출물 파일 이름을 정한다 |
| 다음 실행에서 이어갈 수 없다 | 산출물이 대화에만 남았다 | `artifacts/README.md`와 단계별 파일 산출물을 추가한다 |
| 처음 읽는 사람이 이해하기 어렵다 | 용어 해설이 없다 | Agent, Skill, Orchestrator를 일상 언어로 먼저 설명한다 |
| Codex가 새 스킬을 못 찾는다 | 설치 위치나 세션 문제가 있다 | `.agents/skills` 위치를 확인하고 Codex를 재시작한다 |
| 커스텀 Agent가 기대와 다르게 일한다 | 역할 설명이 넓거나 모호하다 | `.codex/agents/{name}.toml`의 `description`과 `developer_instructions`를 좁힌다 |
| Orchestrator가 모든 일을 직접 한다 | 역할 위임 기준이 없다 | Orchestrator Skill에 단계별 담당자와 산출물 이름을 적는다 |
| Skill이 안 켜진다 | description이 약하다 | 사용자가 말할 법한 트리거 표현을 추가한다 |
| Skill이 엉뚱하게 켜진다 | 제외 조건이 없다 | 쓰면 안 되는 요청을 description에 추가한다 |
| 위험한 행동이 불안하다 | 승인 지점이 없다 | Orchestrator 실패 처리와 승인 조건을 추가한다 |

## 다음 버전으로 반영하기

개선할 때는 한 번에 많이 고치지 않는다. 가장 큰 문제 하나만 고치고 다시 테스트한다.

권장 순서:

1. 실패 사례를 하나 고른다.
2. 실패가 Agent 문제인지, Skill 문제인지, Orchestrator 문제인지 나눈다.
3. 해당 문서 한 곳만 고친다.
4. 같은 테스트를 다시 실행한다.
5. 개선 여부를 기록한다.

## 변경 이력 남기기

기존 하네스를 수정했다면 변경 이력과 개선 기록을 남긴다. 표 형식과 ADR 형식은 `failure-maintenance.md`의 "변경 이력 형식"과 "ADR 형식 (큰 구조 결정)"을 따른다. 변경 이력은 다음 세션에서 하네스가 왜 이렇게 생겼는지 알려 주는 작은 인수인계다.
