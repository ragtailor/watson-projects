# Skill Authoring Guide

이 문서는 하네스 안에서 Skill을 만들거나 점검할 때 읽는다. Skill은 긴 프롬프트가 아니라 반복 업무 매뉴얼이다. 좋은 Skill은 언제 실행되어야 하는지 분명하고, 실행되면 어떤 순서로 일하며, 어떤 형식의 산출물을 남기는지 안정적으로 알려준다.

## Skill의 역할

하네스에서 Skill은 Agent가 따르는 작업법이다.

| 구분 | 역할 |
| --- | --- |
| Agent | 일을 맡는 팀원 |
| Skill | 팀원이 따르는 작업 매뉴얼 |
| Orchestrator Skill | 전체 순서, 전달물, 승인 지점, Agent Team 실행을 묶는 입구 |

Skill은 아래 질문에 답해야 한다.

- 어떤 요청에서 이 Skill을 써야 하는가?
- 어떤 요청에서는 쓰면 안 되는가?
- 먼저 무엇을 읽고 확인해야 하는가?
- 어떤 순서로 작업해야 하는가?
- 결과는 어디에 어떤 형식으로 남겨야 하는가?
- 실패하거나 정보가 부족하면 어떻게 멈추고 물어봐야 하는가?

## 기본 구조

권장 구조는 아래와 같다.

```markdown
---
description: {트리거와 제외 조건이 분명한 한 문단}
---

# {Skill Name}

## 역할

이 Skill이 맡는 반복 업무를 설명한다.

## 사용 조건

- 사용해야 하는 경우
- 사용하지 말아야 하는 경우

## 입력

- 먼저 읽을 파일
- 사용자에게 확인할 정보
- 이전 단계 산출물

## 절차

1. ...
2. ...
3. ...

## 출력

- 저장 위치
- 파일명
- 형식

## 품질 기준

- 통과 기준
- 사람 승인 조건
- 실패 시 처리
```

## Description 작성법

frontmatter의 `description`은 소개문이 아니라 라우팅 조건이다. "이 Skill은 유용합니다"처럼 흐리게 쓰지 말고, 어떤 요청에서 반드시 사용해야 하는지 적는다.

좋은 description은 세 가지를 포함한다.

| 항목 | 설명 |
| --- | --- |
| 트리거 | 사용자의 어떤 표현, 업무, 상황에서 실행할지 |
| 범위 | 이 Skill이 처리하는 산출물과 책임 |
| 제외 조건 | 비슷하지만 이 Skill을 쓰면 안 되는 경우 |

예시:

```yaml
description: 회의 녹취, 메모, 채팅 로그를 바탕으로 회의록, 결정 사항, 액션 아이템을 만들 때 사용한다. 단순 맞춤법 교정이나 이미 완성된 회의록의 문체 수정에는 사용하지 않는다.
```

피해야 할 예시:

```yaml
description: 회의 관련 작업을 도와주는 Skill입니다.
```

이렇게 쓰면 너무 넓어서 다른 Skill과 충돌하기 쉽다.

## frontmatter 작성 기준

Claude Code Skill은 디렉터리명이 호출 이름이 되고, frontmatter는 표시 이름과 트리거 판단을 보조한다. Agent frontmatter의 `name`과 Skill 디렉터리명을 같은 규칙으로 뭉뚱그리지 않는다.

| 항목 | 기준 |
| --- | --- |
| Skill 디렉터리명 | 사용자가 `/skill-name`으로 부르는 호출 이름이다. 소문자·숫자·하이픈을 우선하고, 동작이 보이는 구체적 이름을 쓴다. |
| Skill `name` | 선택값이다. 쓰는 경우 표시 이름으로 생각하고 디렉터리명과 충돌하지 않게 맞춘다. |
| Skill `description` | 권장값이다. **무엇을 + 언제**를 앞부분에 담고, 제외 조건이 있으면 함께 적는다. `description`과 `when_to_use`는 목록에서 길게 잘릴 수 있으므로 핵심을 먼저 쓴다. |
| Agent `name` | Agent 파일의 고유 식별자다. 소문자 영문과 하이픈을 사용하고, 파일명 및 Orchestrator의 호출 이름과 일치시킨다. |

네이밍은 동명사형(gerund)을 우선한다. `processing-pdfs`, `writing-reports`처럼 동작이 보이게 쓰고, `helper`·`utils`·`tools`·`documents`처럼 모호하거나 너무 넓은 이름은 피한다.

`description`은 **3인칭**으로 쓴다. description은 시스템 프롬프트에 주입되므로 "나는 도와줄 수 있다", "당신은 이걸로…" 같은 표현 대신 "회의록을 만들고 액션 아이템을 추출한다"처럼 행동을 서술한다.

## 본문 작성 원칙

Skill 본문은 실행자가 바로 따를 수 있게 쓴다.

- 왜 필요한지 한 번 설명한 뒤 절차로 들어간다.
- 명령형을 사용한다. "할 수 있다"보다 "확인한다", "작성한다", "저장한다"가 좋다.
- 매 실행마다 달라지는 도메인 내용은 예시로만 둔다.
- 파일 경로, 산출물 이름, 승인 조건은 가능한 한 구체적으로 쓴다.
- 긴 배경 설명은 `references/`로 빼고, 본문에는 실행에 필요한 핵심만 둔다.
- 긴 다단계 절차는 체크리스트를 본문에 두고, 실행 시 그 체크리스트를 응답에 복사해 단계마다 체크하도록 안내한다. 검증 단계를 건너뛰는 것을 막는다.

## Progressive Disclosure

Skill 본문에는 자주 쓰는 핵심 절차만 둔다. 세부 기준은 필요할 때 읽는 reference로 분리한다. 로딩은 3단계로 일어나고, 각 단계의 컨텍스트 비용이 다르다.

| 단계 | 로딩 시점 | 토큰 예산 | 넣을 위치 |
| --- | --- | --- | --- |
| 메타데이터 (description 중심, name 선택) | 항상 | 약 100토큰 | frontmatter |
| 본문 | 트리거될 때 | **500줄 미만 / 약 5k 토큰 미만** | `SKILL.md` |
| 리소스 | 필요할 때만 | 사실상 무제한 | `references/`, `scripts/`, `assets/` |

규칙:

- **본문은 500줄 미만**으로 유지한다. 넘어가면 세부를 `references/`로 빼고 본문에는 "언제 이 파일을 읽으라"는 포인터만 남긴다.
- `references/`는 **SKILL.md에서 한 단계 깊이**로 둔다. 참조가 참조를 다시 부르는 깊은 중첩은 피한다. **100줄을 넘는 참조 파일에는 상단에 목차**를 둔다.
- 세 리소스 폴더의 용도는 분명히 다르다.

| 폴더 | 용도 | Claude가 읽나 |
| --- | --- | --- |
| `references/` | 필요할 때 **읽어서** 판단에 쓰는 문서(판단 기준, 예시, 도메인 배경) | 읽음 |
| `scripts/` | bash로 **실행**하는 결정적 코드. 내용이 아니라 출력만 컨텍스트로 들어온다 | 실행 |
| `assets/` | **산출물에 들어가는 재료**(템플릿, 보일러플레이트, 이미지, 폰트, CSS) | 보통 안 읽고 결과물에 채워넣음 |

판단 기준: 매번 필요한 내용은 `SKILL.md`, 가끔 필요한 긴 내용은 `references/`, 손으로 반복하기 귀찮은 절차는 `scripts/`, 결과물 양식·고정 자료는 `assets/`에 둔다. assets는 에이전트가 직접 붙이는 것이 아니라 **Skill이 참조**하고, 그 Skill을 쓰는 에이전트가 거쳐 사용한다.

## 자유도 맞추기

Skill의 지시 강도는 작업이 깨지기 쉬운 정도에 맞춘다. 한 가지 강도로 통일하지 않는다.

| 자유도 | 언제 | 쓰는 방식 |
| --- | --- | --- |
| 높음 | 정답이 여러 개, 판단이 필요 | 텍스트 지침으로 방향만 준다 |
| 중간 | 선호하는 패턴이 있다 | 의사코드나 파라미터화된 절차를 준다 |
| 낮음 | 깨지기 쉽고 일관성이 중요 | "이 스크립트를 그대로 실행한다. 명령을 바꾸지 않는다"처럼 좁힌다 |

비유로는 절벽 사이 좁은 다리(낮은 자유도)와 열린 들판(높은 자유도)이다. 결정적이어야 하는 검증·변환·발송 직전 단계는 좁게, 창작·탐색 단계는 넓게 둔다.

## scripts 작성 기준

반복되는 결정적 절차(링크·숫자·섹션·금지어 점검, 포맷 변환 등)를 `scripts/`로 뺄 때는 아래 품질 기준을 지킨다. 스크립트는 LLM 생성 코드보다 안정적이고 토큰을 아끼지만, 부실하면 오히려 디버깅 비용을 늘린다.

- **solve, don't punt**: 에러를 Claude에게 떠넘기지 말고 스크립트 안에서 처리한다. 파일 없음·권한 없음 같은 예외는 기본값 생성이나 대안 경로로 명시적으로 다룬다.
- **voodoo 상수 금지**: 모든 상수(타임아웃·재시도 횟수 등)에 "왜 이 값인지" 주석을 단다. 작성자가 모르는 값은 Claude도 정할 수 없다.
- **자세한 에러 메시지**: "필드 X 없음. 가능한 필드: a, b, c"처럼 다음 행동을 알려주는 actionable 메시지를 낸다. 불투명한 코드·트레이스백만 던지지 않는다.
- **plan-validate-execute**: 배치·파괴적·고위험 작업은 먼저 계획을 구조화된 파일(예: `changes.json`)로 만들고, 검증 스크립트로 확인한 뒤 실행한다. 검증 가능한 중간 산출물이 오류를 일찍 잡고 되돌리기를 쉽게 한다.
- **실행 의도 명시**: "이 스크립트를 실행한다"인지 "이 알고리즘을 참고로 읽는다"인지 분명히 적는다. 대부분의 유틸리티는 실행이 더 안정적이고 토큰을 아낀다.

## Output Format

하네스용 Skill은 대화 답변만 만들지 않는다. 다음 단계가 이어받을 수 있는 산출물을 남겨야 한다.

출력 형식에는 아래를 포함한다.

- 저장 경로
- 파일명 규칙
- 제목 구조
- 필수 섹션
- 완료 기준
- 다음 단계가 읽어야 할 부분

예시:

```markdown
출력은 `artifacts/02-draft.md`에 저장한다.

필수 섹션:
- 목적
- 대상 독자
- 핵심 메시지
- 초안
- 확인 필요 사항
```

Agent Team에서 쓰는 Skill이라면 어떤 메시지를 팀에 보내야 하는지도 적는다.

```markdown
중대한 누락, 요구사항 충돌, 다음 Agent의 입력 변경이 생기면 Orchestrator와 관련 Agent에게 `SendMessage`로 공유한다. 다음 단계가 반드시 읽어야 하는 내용은 메시지에만 남기지 말고 `artifacts/`에 저장한다.
```

## 예시와 Edge Case

Skill에는 정상 예시뿐 아니라 애매한 예시와 쓰면 안 되는 예시가 필요하다.

| 예시 종류 | 목적 |
| --- | --- |
| 정상 예시 | Skill이 기대대로 실행되는 요청 확인 |
| 애매한 예시 | 사용자에게 질문할 조건 확인 |
| 제외 예시 | 다른 Skill로 넘기거나 직접 답해야 하는 요청 확인 |
| 실패 예시 | 정보 부족, 승인 필요, 위험 작업 처리 확인 |

간단한 테스트 문장:

```markdown
Should trigger:
- "이 회의 녹취로 결정사항과 액션아이템 정리해줘"

Should not trigger:
- "이 문장 맞춤법만 고쳐줘"

Ask before proceeding:
- "지난 회의 정리해줘"처럼 입력 자료가 없는 요청
```

## Orchestrator Skill 작성 시 주의점

Orchestrator Skill은 일반 Skill보다 더 명확해야 한다. 전체 실행을 묶는 입구이기 때문이다.

반드시 포함할 것:

- 하네스 목표
- 읽어야 할 입력과 기존 산출물
- Agent Team 사용 여부
- `TeamCreate`, `TaskCreate`, `TaskUpdate`, `TaskGet`, `SendMessage`, `TeamDelete` 흐름
- Agent별 `tools`, 주요 산출물 계약
- 단계별 산출물 경로
- `artifacts/README.md` 산출물 지도와 최신 상태 갱신 규칙
- with/without 벤치마크를 수행할 때 `light`, `targeted`, `full` 중 두께를 먼저 고르고, `artifacts/evals/iteration-N/{eval-name}/` 아래에 그 두께에 맞는 `prompt.md`, `run-config.md`, 출력 파일, `metrics.json`, `comparison.md`를 저장하는 규칙
- 부분 재실행 시 뒤 단계 산출물을 `stale`로 표시하는 규칙
- 사람 승인 지점
- 최종 산출물의 `사용 가능`, `사람 승인 필요`, `미검증 영역` 표시
- `artifacts/README.md`에서 파일 최신 상태(`current`, `stale`, `needs-review`, `archived`)와 승인 상태(`사용 가능`, `사람 승인 필요`, `미검증 영역 있음`, `해당 없음`)를 분리하는 규칙
- 실패 시 멈춤 조건
- 개선 기록 갱신 조건

Orchestrator Skill 디렉터리명은 `{harness-name}-orchestrator` 형식을 사용한다. frontmatter `name`을 쓴다면 같은 값을 사용한다.

## Skill에 넣지 않을 것

- 모든 배경지식을 본문에 길게 붙여 넣지 않는다.
- 다른 Agent와 Skill의 세부 규칙을 중복해서 복사하지 않는다.
- 위험 작업을 자동 완료하도록 쓰지 않는다.
- "항상 최고 품질로"처럼 검증할 수 없는 기준을 품질 기준으로 쓰지 않는다.
- 파일 산출물 없이 대화 답변만 만드는 흐름으로 끝내지 않는다.
- 시간 민감 정보를 본문에 직접 박지 않는다("2025년 8월 전에는 …" 식). 바뀌는 내용은 "현재 방식"으로 쓰고, 옛 정보는 별도 "old patterns" 섹션으로 분리한다. 그래야 시간이 지나도 본문이 틀리지 않는다.
- 선택지를 과하게 나열하지 않는다. 기본 한 가지를 정하고 예외 경로만 탈출구로 덧붙인다("A를 쓴다. 스캔본일 때만 B"). 여러 대안을 동등하게 늘어놓으면 판단이 흔들린다.

## 점검 체크리스트

Skill을 만들고 나면 아래를 확인한다.

- `description`만 읽어도 언제 쓰는지 알 수 있는가?
- 쓰면 안 되는 경우가 적혀 있는가?
- 입력과 출력이 구체적인가?
