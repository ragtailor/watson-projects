# 하네스 설계 워크플로우

이 문서는 `harness-lab`이 사용자의 일을 하네스 구조로 바꾸기 전에 목표, 자료, 산출물, 검증, 승인 지점을 정리할 때 사용한다. 파일 생성보다 먼저 청사진을 만들고, 사용자가 그 청사진을 승인한 뒤에만 실행 하네스를 구성한다.

## 언제 읽을까

- 사용자가 "하네스를 만들어줘", "에이전트와 스킬을 설계해줘"라고 요청했을 때
- 청사진을 만들기 전에 하네스 7요소와 성숙도를 정리해야 할 때
- 사용자가 원하는 결과가 설명, 청사진, 실제 파일 생성 중 무엇인지 구분해야 할 때
- 하네스가 너무 두껍거나 너무 얇지 않은지 판단해야 할 때

## 청사진 승인 게이트

하네스 생성은 항상 두 단계로 진행한다.

1. 청사진 단계: 파일을 만들지 않고 하네스 7요소, 사람의 작업 절차, 산출물 계약, 실행 모드, Agent 역할표, Skill 목록, Orchestrator 흐름, 테스트 프롬프트를 보여준다.
2. 구성 단계: 사용자가 방금 제시한 청사진에 대해 "좋아, 이 구조로 만들어줘", "이 청사진대로 구성해줘", "승인해, 파일 생성해줘"처럼 명시적으로 승인한 뒤에만 현재 프로젝트에 파일을 만든다.

구성 단계로 넘어가면 안 되는 경우:

- 청사진을 아직 보여주지 않았다.
- 사용자가 첫 요청에서 "바로 만들어줘", "분석 없이 생성해", "파일로 만들어줘"라고만 말했다.
- 사용자가 의견이나 방향만 물었다.

이 경우에는 청사진을 먼저 만들고, 마지막에 구성 여부를 묻는다.

## 하네스 한 문장 정의

하네스는 AI가 덜 추측하고, 더 안전하게, 더 반복 가능하게 일하도록 만드는 작업 환경이다.

좋은 하네스는 AI에게 모든 것을 즉석에서 판단하게 하지 않는다. 목표는 목표로, 자료는 자료로, 반복 절차는 Skill로, 역할은 Agent로, 승인과 검증은 Orchestrator와 체크리스트로 바깥에 꺼내 둔다.

## 숙련도 적응

같은 하네스라도 사용자의 숙련도에 따라 설명의 밀도를 바꾼다. 대화의 단서(쓰는 용어, 질문 수준, 파일·구조 언급 여부)로 수준을 가늠하고 톤을 조절한다. 결과물의 구조는 같고, 설명의 두께만 달라진다.

| 수준 | 단서 | 설명 방식 |
| --- | --- | --- |
| 초보 | 하네스·Agent·Skill 용어가 낯설다, "멋지게/알아서" 같은 막연한 표현 | 비유 먼저, 보편 인테이크 5문항으로 빈칸을 채운 뒤 청사진. 용어는 일상 언어로 먼저 푼다 |
| 실무자 | 산출물·역할·검토를 구체적으로 말한다 | 청사진과 핵심 결정 위주. 비유는 줄이고 역할표·산출물 계약을 바로 보여준다 |
| 전문가 | tools·훅·평가·실행 옵션 같은 용어를 직접 쓴다 | 설계 카드와 스펙 위주로 간결하게. 군더더기 설명을 덜고 트레이드오프만 짚는다 |

초보 수준일 때는 별도 절차 없이 아래 **보편 인테이크 5문항**을 먼저 채운다. 업종·기술과 무관하게 동작한다.

1. 무엇을 만들면 성공인가? (최종 산출물)
2. 누가 읽거나 쓰는가? (독자·사용자)
3. 어떤 자료를 먼저 줄 수 있는가? (입력)
4. 무엇이 틀리면 위험한가? (위험·승인 지점)
5. 한 번 쓰고 끝인가, 반복하는가? (반복성)

이 다섯 답이 산출물 유형, 역할 구성, 두께와 승인 게이트를 거의 결정한다. 수준을 점수처럼 말하지 않고, 부족한 칸만 "확인 필요"로 표시한 뒤 진행한다.

## 하네스 7요소

| 요소 | 쉬운 말 | 확인 질문 | 하네스 산출물 |
| --- | --- | --- | --- |
| 목표 | 무엇을 만들지 정하기 | 최종 결과물은 무엇인가? | 목표 한 문장, 완료 기준 |
| 컨텍스트 | AI가 봐야 할 자료 | 어떤 문서와 정보를 먼저 줄 것인가? 전부 적재할까, 경로만 주고 찾게 할까? | 참고 자료 목록, 입력 폴더, (대용량은) 경로·검색 식별자 |
| 도구 | AI가 사용할 손과 발 | 어떤 도구가 필요하고 어디까지 허용할 것인가? | Agent 도구 범위, 읽기/쓰기 기준 |
| 중간 산출물 | 바로 최종으로 가지 않는 중간 결과 | 초안 전에 무엇을 먼저 확인할 것인가? | 요약표, 매트릭스, 초안, 검토표 |
| 검증 | 좋은 결과의 기준 | 무엇을 만족해야 괜찮은 결과인가? | 체크리스트, 테스트 프롬프트, 루브릭 |
| 권한과 승인 | 사람이 책임질 지점 | 발송, 게시, 제출, 삭제 전에 누가 확인하는가? | 승인 조건, 멈춤 조건 |
| 기록과 개선 | 다음번을 위한 기억 | 무엇을 저장하고 다음번에 고칠 것인가? | 개선 기록, 변경 이력 |

컨텍스트는 무조건 많이 넣는 게 아니라 **유한 자원**으로 다룬다. 토큰이 길어질수록 성능이 떨어지므로(context rot), 자료가 작으면 미리 적재하되, 코드베이스·대용량 문서처럼 클 때는 전부 적재하지 말고 **경량 식별자(파일 경로·URL·grep/glob 검색)** 를 주고 에이전트가 런타임에 필요한 부분만 가져오게 한다(just-in-time 검색). 미리 적재가 stale해지는 문제를 피하고 컨텍스트 오염을 줄인다. 속도가 필요하면 일부만 미리 적재하고 나머지는 런타임 검색으로 두는 하이브리드도 쓴다.

## 하네스 성숙도 5단계

| 단계 | 이름 | 상태 | 다음 보강 |
| --- | --- | --- | --- |
| 1단계 | 요청만 있는 상태 | 프롬프트는 있지만 기준과 검증이 없다 | 목표, 대상, 참고 자료, 금지 조건을 적는다 |
| 2단계 | 목표와 자료가 있는 상태 | 무엇을 해야 하는지와 참고 자료가 있다 | 중간 산출물과 질문 목록을 둔다 |
| 3단계 | 중간 산출물과 검증이 있는 상태 | 초안 전에 기준표와 검토표가 있다 | 실패 처리와 사람 승인 조건을 정한다 |
| 4단계 | 실패 처리와 인수인계가 있는 상태 | 재시도, 기록, 다음 작업 인계가 가능하다 | 실행 기록과 개선 루프를 둔다 |
| 5단계 | 운영과 개선 루프가 있는 상태 | 승인, 메모리, 지표, 개선 루프가 함께 돈다 | 오래된 규칙과 메모리를 정리한다 |

진단할 때는 점수 매기듯 말하지 않는다. "지금은 2단계에 가까워서, 중간 산출물 하나와 검증 기준 세 개를 붙이면 바로 좋아집니다"처럼 다음 행동으로 연결한다.

## Phase 0-7 흐름

| --- | --- | --- | --- |
| 0 | 현재 상황 확인 | 무엇을 만들고, 누가 사용할까? | 목표, 경험 수준, 성숙도, 기존 파일 점검 |

| 2 | 산출물 정의 | 무엇이 나오면 성공일까? | 하네스 7요소, 산출물 계약, 성공 기준 |
| 3 | 실행 모드와 Agent Team 패턴 선택 | 혼자 할까, 격리 위임할까, 팀으로 조율할까? | 단일 흐름, Subagent, Agent Team 중 선택 |

| 6 | Orchestrator 설계 | 누가 순서와 전달을 관리할까? | 실행 흐름, 산출물 전달, 오류 처리 |
| 7 | 테스트와 개선 | 제대로 작동했는지 어떻게 볼까? | 테스트 프롬프트, 변경 이력, 개선 기록 |

## Phase 0 분기

Phase 0에서는 먼저 현재 프로젝트의 상태를 읽고, 신규 구축인지 기존 확장인지 운영/유지보수인지 구분한다.

확인할 위치:

- `CLAUDE.md`
- `.claude/agents/`
- `.claude/skills/`
- `artifacts/`

| 분기 | 판단 기준 | 다음 행동 |
| --- | --- | --- |
| 신규 구축 | `.claude/agents`, `.claude/skills`, 관련 `CLAUDE.md` 포인터가 없거나 비어 있다 | Phase 1부터 전체 청사진을 만든다 |
| 기존 확장 | 기존 하네스가 있고 Agent, Skill, Task, 산출물만 추가하거나 바꾸려 한다 | 기존 구조를 읽고 필요한 Phase만 다시 설계한다 |
| 운영/유지보수 | 점검, 감사, 동기화, drift, 오래된 규칙 정리를 요청한다 | 테스트, QA, 유지보수 가이드를 기준으로 개선 청사진을 만든다 |

### 기존 확장 매트릭스

기존 확장 시에는 아래처럼 필요한 Phase만 다시 본다.

| 변경 유형 | 다시 볼 Phase |
| --- | --- |
| Agent 추가 | Phase 3, Phase 4, Phase 6, Phase 7 |
| Skill 추가/수정 | Phase 5, Phase 6, Phase 7 |
| Orchestrator 흐름 변경 | Phase 3, Phase 6, Phase 7 |
| 산출물 구조 변경 | Phase 2, Phase 6, Phase 7 |

기존 확장도 바로 파일을 고치지 않는다. 먼저 현재 구조와 바꿀 부분을 요약하고, 개선 청사진을 보여준 뒤 사용자가 적용을 요청하면 수정한다.

## Phase 1. 도메인 분석

도메인 분석은 "이 일이 어떤 일인가"를 읽는 단계다. 역할 분리, 사람 승인 지점, 적합성 판정이 모두 여기서 나오므로 하네스 설계에서 가장 중요한 분석이다. 먼저 "사람이 한다면 어떤 순서로 하는가"를 일상 언어로 풀어쓰며 시작하고(예: 블로그 글 = 주제 정함 → 자료 모음 → 목차 → 초안 → 검토), 그 위에서 아래를 함께 읽는다.

| 분석 항목 | 무엇을 보나 | 무엇을 결정하나 |
| --- | --- | --- |
| 작업 유형 | 생성·검증·편집·분석·조사 중 무엇인가 | 팀 패턴 선택 |
| 기존 자료·코드베이스 | 코드면 기술 스택·데이터 모델·주요 모듈, 문서 업무면 기존 양식·자료·입력 형식·제약 | 컨텍스트, 재사용 가능 범위 |
| 반복 보조 작업 | 매번 반복되는 조사·정리·검토 | 역할(Agent) 후보 |
| 위험 지점·외부 의존 | 외부 발송·제출·삭제·결제·개인정보·법적 표현 | 사람 승인 지점, 하네스 두께 |
| 기존 하네스 충돌 | 기존 Agent·Skill과 겹침, drift | 신규/확장/운영 분기 |
| 숙련도·맥락 단서 | 사용자가 쓰는 용어, 질문 수준 | 설명 밀도 |

이 단계에서 Agent·Skill이라는 말을 너무 빨리 꺼내지 않는다. 먼저 사람의 절차와 위 항목을 정리한 뒤, 그 결과를 다음 단계의 입력으로 넘긴다. 특히 도메인 분석은 적합성 게이트의 입력이다 — 반복성·구조 안정성·검증 가능성·외부 신호를 여기서 읽어 "이 업무가 하네스화할 가치가 있는가"를 판정한다. 코드 프로젝트뿐 아니라 일상 문서·운영 업무에도 똑같이 적용한다(코드베이스 대신 기존 문서·양식·반복 절차를 본다).

## Phase 2. 산출물 정의

마지막에 무엇이 나오면 성공인지 먼저 정한다. harness-lab는 산출물 우선이라, 이 단계가 뒤 설계를 거꾸로 끌고 간다.

- 산출물 유형을 고른다(문서형·시각형·운영형·반복형, 조합 가능).
- 성공 기준과 실패 기준을 적는다(내용 정확성·누락·사용 가능성·독자 수준).
- 산출물 계약을 잡는다: 어떤 파일을, 누가, 어떤 형식으로 만들고, 다음 단계가 어떻게 읽는지. 상세는 `orchestrator-design.md`.
- **평가를 먼저 정의한다(평가 우선)**: 문서를 많이 쓰고 파일을 만들기 전에, 성공/실패 기준을 통과·탈락이 갈리는 평가 시나리오 2~3개로 구체화하고 가능하면 무하네스 baseline을 먼저 잡는다. 이 평가가 설계를 끌고 가서, 통과에 필요한 **최소 구조**만 만들게 한다(상상한 요구가 아니라 실제 실패를 메운다). 여기서 만든 평가 시나리오와 baseline은 Phase 7 테스트·with/without 비교로 그대로 이어진다.
- 외부 발송·제출 같은 위험 산출물이면 사람 승인 지점을 여기서 표시한다.

출력: 산출물 유형, 성공/실패 기준, 산출물 계약 초안.

## Phase 3. 실행 모드와 Agent Team 패턴 선택

도메인 분석과 산출물을 보고 협업 구조를 고른다.

1. 실행 모드: 단일 흐름 → Subagent → Agent Team 순으로, 조율 비용과 문맥 경계를 보고 필요한 최소를 고른다.
2. 패턴: Pipeline·Fan-out/Fan-in·Expert Pool·Producer-Reviewer·Supervisor·Hierarchical 중 작업에 맞는 것. 위험·합격 기준이 분명하면 반복 검증 루프·투표·Reflection 같은 동적 제어 패턴도 고려.
3. 분리 기준: 문맥 경계·전문성·입출력 독립·재사용·위험 4축으로 Agent 분리 여부를 판단한다.

상세는 `agent-design.md`(실행 모드 선택, 팀 패턴, Agent를 분리하기 전 질문, 반복·합의·자기비판 패턴).

출력: 실행 모드, 팀 패턴, 분리할 역할 후보.

## Phase 4. Agent 설계

누가 어떤 역할을 맡을지 정한다.

- 각 Agent: 책임·입력·출력·하지 말아야 할 일.
- frontmatter: `name`·`description`·`tools`(최소 권한)·`model`(역할별 루브릭 — 규칙기반=haiku, 의미해석=sonnet, 상충·구조=opus, 장시간·검증=fable). `type` 필드는 없다 — 빌트인은 `subagent_type`로 지정하되 빌트인이라도 `.md` 파일을 만든다.
- 생성자와 평가자를 분리하고, 평가는 외부 신호(테스트·체크리스트·사람 승인)에 묶는다.
- QA·Reviewer는 존재 확인보다 경계면 교차 검증을 우선하고, 전체 후 1회가 아니라 중요한 중간 산출물 직후 점진적으로 실행한다.

상세는 `agent-design.md`(Agent frontmatter 설계, Agent 모델 선택, 빌트인 규칙, 생성자-평가자 분리).

출력: Agent 역할표(이름·tools·model·책임), 역할 선정 이유.

## Phase 5. Skill 설계

각 역할이 따를 작업 매뉴얼을 만든다.

- 각 Skill: 트리거·절차·출력 형식·품질 체크·예외.
- description은 "무엇을 + 언제"를 3인칭으로, should/should-not 경계를 분명히 한다.
- 본문은 lean하게, 500줄 넘으면 세부를 `references/`로 분리(progressive disclosure). 반복되는 결정적 절차는 `scripts/` 후보로 본다.

상세는 `skill-authoring-guide.md`.

출력: Skill 목록(트리거·절차·연결 Agent).

## Phase 6. Orchestrator 설계

역할들의 순서·전달·실패 처리를 묶는다.

- 실행 흐름: 입력 확인 → 단계별 담당 → 중간 산출물 파일 → 통합 → 승인 게이트 → 정리.
- 데이터 전달: 메시지(조율)·태스크(상태)·파일(기억)·반환값(단발) 중 선택. Agent Team이면 `TeamCreate`~`TeamDelete` 계약. 상세는 `orchestrator-design.md`.
- **드라이런(설계 검증)**: 파일을 만들기 전에 흐름 그래프를 점검한다 — (a) Phase 순서가 논리적인가, (b) 각 단계의 입력이 이전 단계의 출력과 실제로 맞물리는가(끊긴 dead-link 없는가), (c) 에러·미승인 시 폴백 경로가 실행 가능한가. 끊긴 구간이 있으면 산출물 계약을 고친 뒤 진행한다.

출력: Orchestrator 흐름, 데이터 전달 방식, 드라이런 점검 결과.

## Phase 7. 테스트와 개선

제대로 작동하는지 보고 다음 버전으로 잇는다.

- 테스트 3종(정상·애매·실패 위험) + 실무면 부정·반복. 결정적 검증 단계가 있으면 결함 주입 회귀를 기본으로.
- 트리거 검증: should-trigger와 near-miss should-not-trigger.
- 가치 증명: 벤치마크 두께(`light`/`targeted`/`full`)로 with/without 비교.
- **피드백 → 수정 대상**: 피드백을 받으면 유형별로 어디를 고칠지 정한다.

| 피드백 유형 | 수정 대상 |
| --- | --- |
| 결과물 품질이 낮다 | 해당 역할의 Skill |
| 역할이 빠졌다/겹친다 | Agent 정의 또는 Orchestrator 팀 구성 |
| 순서가 어색하다 | Orchestrator 흐름 |
| 안 켜진다/엉뚱하게 켜진다 | Skill·Orchestrator description |
| 같은 실패가 반복된다 | 테스트·결함 주입 회귀 세트 |

- **진화 트리거(선제 제안)**: 사용자가 "고쳐줘"라고 말하기 전에도 아래 신호를 보면 개선을 먼저 제안한다 — (a) 같은 유형의 피드백이 2회 이상 반복, (b) 특정 Agent·단계가 반복적으로 실패, (c) 사용자가 Orchestrator를 우회해 수동으로 처리하는 것이 관찰됨.
- 덜어내기 진단으로 과잉 조각을 줄이고, 큰 구조 결정은 ADR로 남긴다. 상세는 `testing-qa-evolution.md`.

출력: 테스트 결과, 개선 기록, 필요 시 ADR.

## 하네스 적합성 게이트

하네스를 만들기 전에 "이 업무가 하네스로 만들 가치가 있는가"를 먼저 판정한다. 만들기 쉽다고 다 만들면, 일회성·고변동·전문가 업무에서는 오히려 손이 더 가거나 느려진다. 실제로 숙련자가 성숙한 작업에 AI를 붙였을 때 본인은 빨라졌다고 느끼지만 측정해 보면 더 느려진 사례가 있다. 체감과 실제가 다를 수 있으므로, 진입부에서 아래 6차원을 보고 이득 쪽이 분명할 때만 두껍게 만든다.

| 차원 | 하네스가 이득 | 손해(얇게/생략) | 확인 질문 |
| --- | --- | --- | --- |
| 반복성 | 같은 절차를 자주 반복 | 일회성·매번 다른 일 | 이 일을 앞으로도 반복하는가? |
| 구조 안정성 | 흐름을 미리 그릴 수 있다 | 매번 즉흥적으로 달라진다 | 단계와 분기를 사전에 매핑할 수 있는가? |
| 검증 가능성 | 정답·체크리스트로 값싸게 검증 | 검증이 생성만큼 어렵다 | 좋은 결과인지 객관적으로 가릴 수 있는가? |
| 외부 신호 | 테스트·기준·사람 승인이 있다 | 순수 자기평가에만 의존 | 검증을 무엇에 기댈 수 있는가? |
| 숙련도 | 저숙련·표준화 업무에 절차가 도움 | 고숙련·고맥락 전문가 업무 | 누가 하든 같은 절차가 도움이 되는가? |
| 체인 길이 | 짧고 검증 게이트가 있다 | 긴 다단계 자율 체인 | 단계가 길어 오류가 누적되지 않는가? |

판정은 점수가 아니라 방향이다. 이득 쪽이 분명하면 두껍게, 손해 신호가 많으면 단일 흐름·체크리스트로 얇게 가거나 하네스를 만들지 않는다. 가치가 애매하면 with/without 비교(`light`)로 먼저 확인한 뒤 결정한다. 효율은 "에이전트를 많이 둬서"가 아니라 구조화·검증·승인 게이트에서 나오므로, 멀티에이전트 자체를 목표로 삼지 않는다.

## 한 장 설계 카드

청사진 전에 아래 항목을 채운다. 모든 항목이 비어 있으면 먼저 질문하고, 절반 이상 채워져 있으면 남은 빈칸을 "확인 필요"로 표시한 뒤 청사진을 만들 수 있다.

| 항목 | 적을 내용 |
| --- | --- |
| 1. 반복해서 하는 일 | 어떤 일을 하네스로 만들고 싶은가? |
| 2. 흔들리는 이유 | 결과가 매번 달라지는 이유는 무엇인가? |
| 3. 최종 결과물 | 마지막에 어떤 형태의 결과가 나와야 하는가? |
| 4. 먼저 줄 자료 | 문서, 메모, 링크, 기준표, 예시 중 무엇을 줄 것인가? |
| 5. 먼저 물어볼 질문 | 정보가 부족할 때 AI가 무엇을 물어봐야 하는가? |
| 6. 중간 산출물 | 최종 결과 전에 어떤 표, 목록, 초안, 체크리스트가 필요한가? |
| 7. 검증 기준 | 무엇을 만족해야 좋은 결과라고 볼 수 있는가? |
| 8. 실패 처리 | 다시 시도할 것인가, 질문할 것인가, 멈출 것인가? |
| 9. 사람 승인 지점 | 어떤 순간에는 반드시 사람이 확인해야 하는가? |
| 10. 다음 기록 | 다음 작업을 위해 무엇을 저장하거나 기억해야 하는가? |
| 11. Agent 권한 | 각 역할에 어떤 `tools`가 필요한가? |
| 12. 핵심 결정(ADR) | 큰 구조 결정은 무엇이고, 버린 대안·가정·트레이드오프·재검토 조건은 무엇인가? (작성 형식은 `testing-qa-evolution.md`) |

## 하네스 두께 정하기

| 상황 | 적당한 두께 | 이유 |
| --- | --- | --- |
| 아이디어 브레인스토밍 | 얇은 하네스 | 자유로운 탐색이 더 중요하다 |
| 개인 메모 정리 | 얇은 하네스 | 실패 비용이 낮다 |
| 팀 회의록 공유 | 중간 하네스 | 누락과 오해를 줄여야 한다 |
| 고객 답변 | 중간에서 두꺼운 하네스 | 외부 영향과 브랜드 위험이 있다 |
| 사업계획서 제출 | 두꺼운 하네스 | 평가 기준, 증빙, 제출 책임이 있다 |
| 결제, 삭제, 배포, 법적 판단 | 매우 두꺼운 하네스 | 되돌리기 어렵고 책임이 크다 |

두께는 위험도이고, 두께가 곧 켜지는 검증과 사람 승인의 양이다. 한 축으로 묶어 자동으로 맞춘다.

| 두께 | 켜지는 rigor |
| --- | --- |
| 얇음 | frontmatter 위생 + 산출물 파일 + 기본 테스트 |
| 중간 | 위 + 검토 역할 + 발송 전 사람 승인 + A/B·near-miss 검증 |
| 두꺼움 | 위 + 증거 기반 검증 + 사람 승인 강화 + 미검증 영역 명시 |

특정 업종에 맞춘 고정 구성을 미리 넣지 않는다. 사용자가 요청한 일에서 위험 신호(외부 발송·제출·결제·삭제·개인정보·법적 표현)를 읽어 두께와 사람 승인 지점을 그때 정한다.

## 긴 작업: 컨텍스트 리셋과 핸드오프

여러 단계가 길게 이어지는 하네스에서는 한 대화에 모든 맥락을 쌓다가 컨텍스트가 가득 차는 문제가 생긴다. 컨텍스트가 길어지면 모델은 작업을 조기에 끝내려 하거나(컨텍스트 불안), 앞쪽 정보를 흐리게 다룬다. 이때 요약으로 계속 이어붙이는 것(compaction)은 연속성은 주지만 깨끗한 시작은 주지 못한다.

긴 작업은 sprint·Phase 단위로 끊고, 단계 사이에 핸드오프 산출물을 남긴 뒤 깨끗한 컨텍스트로 다시 시작하는 방식을 기본으로 둔다.

| 방식 | 언제 | 한계 |
| --- | --- | --- |
| 요약 이어붙이기 | 짧은 작업, 맥락이 작을 때 | 길어지면 누적 노이즈와 컨텍스트 불안이 커진다 |
| 컨텍스트 리셋 + 핸드오프 | 단계가 많고 길거나, 팀을 바꿀 때 | 핸드오프 산출물을 잘 써야 한다 |

핸드오프 산출물(`artifacts/handoff.md` 등)에 담을 것:

- 지금까지의 결정과 그 근거
- 다음 단계가 그대로 읽어야 할 입력 파일 경로
- 남은 할 일과 아직 검증되지 않은 영역
- 사람 승인이 걸린 지점과 현재 승인 상태

이렇게 하면 다음 단계나 다음 팀은 전체 대화를 다시 읽지 않고 핸드오프 산출물과 필요한 산출물만 읽어 깨끗하게 이어받는다. 단계마다 착수 전에 "이 단계가 끝나면 무엇이 있어야 하는가"를 성공 계약으로 먼저 정해, 리셋 후에도 기준이 흔들리지 않게 한다. Agent Team에서 팀을 바꿀 때의 구체 절차는 `references/orchestrator-design.md`의 팀 재구성 규칙을 따른다.

## 청사진 입력 프롬프트

```text
/harness-lab 아래 설계 카드를 바탕으로 현재 프로젝트에서 사용할 하네스를 설계해줘.

1. 반복 업무:
2. 먼저 줄 자료:
3. AI가 먼저 물어봐야 할 질문:
4. 중간 산출물:
5. 검증 기준:
6. 실패 처리와 사람 승인:
7. 다음 개선 기록:

먼저 청사진을 보여주고, 내가 승인하면 실제로 사용할 수 있는 agents와 skills를 만들어줘.
```

## 청사진에 반드시 반영할 것

- 최종 결과물과 중간 산출물을 구분한다.
- AI가 추측하면 안 되는 부분을 "먼저 질문"으로 빼둔다.
- Agent는 필요한 만큼만 만든다.
- Agent 역할표에는 각 Agent의 `tools`를 포함한다.
- Skill은 반복 절차가 있는 역할에만 만든다.
