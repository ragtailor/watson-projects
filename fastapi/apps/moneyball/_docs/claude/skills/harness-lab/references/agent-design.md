# Agent Team 설계 가이드

이 문서는 Agent, Skill, 실행 모드, Agent Team 패턴을 함께 설계할 때 사용한다. 핵심은 "누가 하는가"를 분명히 하고, Agent 파일을 어떻게 설계·분리하며 어떤 팀 패턴으로 묶을지 정하는 것이다. Orchestrator 런타임(실행 계약·흐름·Task·데이터 전달·에러)은 `orchestrator-design.md`를 본다.

## 목차

1. 언제 읽을까
2. Agent, Skill, Orchestrator의 차이
3. Agent frontmatter 설계, 모델 선택, 빌트인 규칙
4. Agent를 분리하기 전 질문
5. 실행 모드 선택
6. Agent 파일 구조
7. 팀 패턴, 팀 크기
8. 생성자-평가자 분리, 반복·합의·자기비판 패턴, Agent Team 안티패턴

## 언제 읽을까

- Agent를 몇 개 만들지 판단해야 할 때
- 단일 흐름, Subagent, Agent Team 중 무엇이 맞는지 정해야 할 때
- Agent 파일에 팀 통신 프로토콜을 작성해야 할 때

## Agent, Skill, Orchestrator의 차이

| 요소 | 쉬운 비유 | 담는 내용 | 프로젝트 위치 |
| --- | --- | --- | --- |
| Agent | 일을 맡은 팀원 | 역할, 책임, 입력, 출력, 도구 범위, 하지 말아야 할 일 | `.claude/agents/{name}.md` |
| Skill | 업무 매뉴얼 | 트리거, 절차, 출력 형식, 품질 기준, 예외 처리 | `.claude/skills/{name}/SKILL.md` |
| Orchestrator | 팀장과 진행표 | 순서, 담당자, 중간 산출물, 실패 처리, 승인 조건 | `.claude/skills/{harness-name}-orchestrator/SKILL.md` |

좋은 하네스는 담당자 카드와 업무 매뉴얼을 분리한다. Agent 파일에 모든 절차를 길게 넣으면 재사용이 어렵고, Skill에 모든 역할 정보를 넣으면 누가 책임지는지 흐려진다.

## Agent frontmatter 설계

Agent 파일의 frontmatter에는 역할과 실행 범위를 먼저 고정한다. 특히 `tools`는 기능별로 명시해두면 팀 실행 중 권한 경계가 흔들리지 않는다. `model`은 역할의 추론 깊이에 맞춰 아래 "Agent 모델 선택" 루브릭으로 고른다. 직접 Subagent와 Agent Team teammate는 모델 정책이 다를 수 있으므로, 팀 실행에서는 팀 생성 지시와 실행 기록에 실제 모델 정책을 남긴다.

| 필드 | 기준 |
| --- | --- |
| `name` | 소문자와 하이픈을 쓰고, 파일명과 맞춘다. Orchestrator가 부르는 이름도 이 값과 일치해야 한다. |
| `description` | 언제 이 Agent를 써야 하는지 구체적으로 적는다. 소개문이 아니라 위임 조건이다. |
| `tools` | Agent가 실제로 필요한 도구만 허용한다. 생략하면 상위 세션의 도구를 넓게 상속하므로 실무 하네스에서는 명시를 우선한다. |
| `disallowedTools` | 대부분의 도구는 상속하되 일부만 막아야 할 때 쓴다. |
| `skills` | 직접 Subagent로 호출할 때 보조적으로 쓸 수 있다. Agent Team teammate 실행에서는 필수 매뉴얼 전달 수단으로 의존하지 않는다. |
| `model` | 역할에 맞춰 모델 선택 루브릭으로 고른다. 별칭 `haiku`/`sonnet`/`opus`/`fable`, 풀 모델 ID, `inherit`(생략 시 기본)을 쓴다. frontmatter에 `type` 필드는 없다. |

역할별 기본 추천:

| 역할 | 추천 tools | 이유 |
| --- | --- | --- |
| 자료 조사, 요구사항 정리 | `Read, Grep, Glob` | 읽기와 판단 중심이며 쓰기 권한이 꼭 필요하지 않다. |
| 초안 작성, 문서 구성 | `Read, Grep, Glob, Write, Edit` | 산출물 작성이 필요하므로 제한된 쓰기 권한을 준다. |
| 구현 담당 | `Read, Grep, Glob, Write, Edit, Bash` | 코드 수정과 검증 명령이 필요하다. |
| QA, Reviewer | `Read, Grep, Glob, Bash, Write` | 교차 검증과 테스트 실행이 필요하다. 검증 대상은 수정하지 않고 자기 판정 파일만 `Write`로 남긴다(본문에 쓰기 대상을 자기 산출물로 한정). `Edit`은 주지 않아 대상을 직접 못 고치게 한다. |
| 최종 통합, 위험 판단 | 작업 성격에 맞게 제한 | 상충 결과를 통합하거나 승인 조건을 판단한다. |
| 단순 분류, 포맷 정리 | `Read, Grep, Glob` | 위험도가 낮아도 불필요한 쓰기 권한은 주지 않는다. |

`tools`는 편의를 위해 넓히는 값이 아니라 책임 경계다. 단, **산출물을 파일로 남기는 모든 역할은 `Write`가 필요하다** — "최소 권한"은 "쓰기 금지"가 아니라 "쓰기 대상을 자기 산출물로 한정"하는 것이다. 위 표에서 `Write`가 없는 행은 자기 산출물조차 파일로 남기지 않는 경우에 한하며, 파일 산출물이 있으면 `Write`를 더한다. 핵심 구분은 **대상 수정 권한(`Edit`)과 자기 산출물 작성(`Write`)을 분리**하는 것이다. Reviewer에게 `Edit`을 주면 검토자가 검증 대상을 직접 고쳐 작성자·검토자 책임이 섞이지만, 자기 판정 파일을 남기는 `Write`는 그와 다르므로 검토자에게도 준다. `Write` 없는 Agent에게 파일 산출을 맡기면 저장에 실패하거나, 저장한 척 본문 요약으로 대체하는 실패가 난다.

## Agent 모델 선택

`model`은 역할의 추론 깊이에 맞춰 고른다. 전 에이전트에 opus를 박는 것(과지출)도, 무조건 생략하는 것(티어 무시)도 피한다. 균형 작업이면 `sonnet`으로 두거나 생략(=inherit)한다.

| 작업 성격 | model | 이유 |
| --- | --- | --- |
| 규칙 기반·정적 점검·추출·분류·고볼륨 | `haiku` | 빠르고 저비용, 의미 해석이 거의 필요 없다 |
| 의미 해석·분석·리뷰·일반 작성(균형) | `sonnet` | 능력과 속도의 균형, 대부분의 역할 기본값 |
| 상충 해소·구조 판단·설계·리팩토링·복잡 추론 | `opus` | 깊은 추론과 아키텍처 결정 |
| 장시간 자율 세션·다단계 검증·착수 전 조사 | `fable` | 긴 세션 유지와 자가 검증 |

예: 정적 점검·필드 추출 워커는 `haiku`, 보안 감사·코드 리뷰처럼 의미 해석이 필요한 역할은 `sonnet`, 설계·리팩토링처럼 상충 해소가 필요한 역할은 `opus`. Agent Team에서는 통합·위험 판단을 맡는 리드에 강한 모델을, 병렬 조사·정적 점검 워커에 가벼운 모델을 섞는다. 값은 별칭(`haiku`/`sonnet`/`opus`/`fable`), 풀 모델 ID, `inherit` 중 하나이며, 생략하면 세션 모델을 상속한다.

**한 팀에 같은 모델을 일괄로 박지 않는다.** 모델은 팀 단위가 아니라 역할 단위로 고른다. 같은 하네스 안에서도 역할마다 추론 깊이가 다르므로 티어가 갈린다. 예를 들어 코드 리뷰 팀이라면:

| 역할 | model | 이유 |
| --- | --- | --- |
| review-lead (통합·심각도 종합) | `opus` | 상충 결과 통합, 우선순위 판단 |
| review-architect (의존성·결합도) | `opus` | 구조 판단 |
| review-security (취약점 의미 해석) | `sonnet` | 위험도 높으면 `opus`로 올림 |
| review-performance (병목 분석) | `sonnet` | 의미 해석 리뷰 |
| review-style (네이밍·데드코드·길이) | `haiku` | 규칙 기반 정적 점검 |

전부 `opus`(과지출)나 전부 생략(티어 무시)은 둘 다 실패다. 팀을 배치 생성할 때 한 값을 모든 팀원에 복사하지 말고, 위 루브릭으로 역할마다 따로 정한다.

**effort를 opus 점프의 대안으로 먼저 본다.** `sonnet`으로 부족해 보일 때 바로 `opus`로 올리기 전에, `sonnet` + 높은 reasoning effort(`effort: high` 등)가 더 싸게 깊이를 줄 수 있는지 본다. 모델 티어 상향과 effort 상향은 별개 레버이고, 둘을 함께 조절한다. effort는 frontmatter 옵션이며 기본은 생략(상속)이다.

**Agent Team 모드에서는 frontmatter `model`이 런타임을 보장하지 못할 수 있다.** 직접 Subagent 호출은 `.md`의 `model`을 따르지만, Agent Team teammate의 실제 모델은 `TeamCreate` 지시나 팀 기본 설정으로 정해질 수 있다. 그래서 팀 하네스에서는 (1) 각 Agent 파일의 `model`을 역할 루브릭으로 정하고, (2) Orchestrator의 `TeamCreate` 단계와 벤치마크 `run-config.md`에 같은 모델 정책을 명시해 둘을 일치시킨다. frontmatter만 고치고 팀 생성 지시를 비워 두면, 적은 값과 실제 실행 모델이 어긋난다.

**배정한 티어로 테스트한다.** 모델은 스킬·에이전트 지침의 효과를 좌우하므로, 지정한 그 티어로 실제 확인한다 — `haiku`를 준 역할은 그 지침이 haiku에도 충분한지(설명을 더 줘야 하는지), `opus`를 준 역할은 과설명으로 토큰을 낭비하지 않는지 본다. 한 티어에서 잘 도는 지침이 다른 티어에서는 부족하거나 과할 수 있다.

## 빌트인 타입과 Agent 파일 규칙

frontmatter에는 `type` 필드가 없다. 빌트인 타입(`general-purpose`, `Explore`, `Plan`)은 Agent 도구의 `subagent_type` 파라미터로 지정한다. 빌트인 타입을 쓰더라도 `.claude/agents/{name}.md` 정의 파일을 만들어 역할·원칙·프로토콜을 담는다. Agent 도구의 prompt에 역할을 직접 inline하지 않는다 — 파일로 있어야 다음 세션에서 재사용되고 팀 협업 품질이 보장된다.

## Agent를 분리하기 전 질문

Agent는 많을수록 좋은 것이 아니다. 아래 질문 중 두 개 이상이 "예"일 때 분리를 검토한다.

| 질문 | 분리해도 좋은 신호 | 분리하지 않는 편이 나은 신호 |
| --- | --- | --- |
| 문맥 경계가 분명한가? | 서로 다른 자료를 봐도 된다 | 같은 정보를 계속 이어서 봐야 한다 |
| 전문성이 다른가? | 조사, 재무, 편집, 법무처럼 판단 기준이 다르다 | 같은 산출물을 조금씩 나눠 쓰는 정도다 |
| 입력과 출력이 독립적인가? | 결과만 넘겨도 다음 단계가 쓸 수 있다 | 중간 판단을 계속 공유해야 한다 |
| 반복 재사용할 책임인가? | 다른 하네스에서도 자주 부를 역할이다 | 이번 한 번만 필요한 작은 단계다 |
| 빠지면 위험한가? | 검증, 승인, 근거 확인처럼 품질이나 안전에 영향이 크다 | Orchestrator 체크리스트로 충분하다 |

역할 이름만 보고 나누지 않는다. 같은 정보를 계속 이어서 해석해야 하면 한 Agent가 맡고, 독립적으로 조사하거나 다른 전문 기준으로 검토해야 할 때만 나눈다.

## 실행 모드 선택

Agent Team-first는 "무조건 팀을 만든다"는 뜻이 아니다. 기본 후보를 Agent Team으로 두되, 조율 비용과 문맥 경계를 보고 최종 모드를 고른다.

| 질문 | 단일 흐름 | Subagent | Agent Team |
| --- | --- | --- | --- |
| 작업이 작고 한 사람이 처리해도 되는가? | 적합 | 과함 | 과함 |
| 독립 조사 후 결과만 합치면 되는가? | 가능 | 적합 | 보통 과함 |
| 중간 발견이 다른 역할의 판단을 바꾸는가? | 부족 | 부족 | 적합 |
| 공유 작업 목록과 진행 상태 추적이 필요한가? | 부족 | 부족 | 적합 |
| 생성자와 검토자가 실시간으로 의견을 주고받아야 하는가? | 부족 | 제한적 | 적합 |
| 여러 역할이 같은 파일을 동시에 고쳐야 하는가? | 순차 권장 | 주의 | 팀 안에서도 파일 소유권 분리 필요 |

판단 순서:

1. 단일 흐름으로 해결 가능한지 본다.
2. 문맥이 너무 커지거나 독립 조사가 많으면 Subagent를 고려한다.
3. 중간 상태를 계속 맞춰야 하면 Agent Team으로 간다.
4. 팀으로 가더라도 동시에 같은 파일을 수정하지 않게 소유권을 나눈다.
5. 비용, 속도, 품질이 실제로 좋아질지 테스트 기준을 둔다.

단순한 것에서 시작해 필요할 때만 복잡도를 올린다. 멀티에이전트는 공짜가 아니다. 여러 에이전트가 각자 컨텍스트를 돌리면 단일 흐름보다 토큰을 훨씬 많이(경험적으로 채팅 대비 십수 배) 쓴다. 그래서 팀은 **가치가 높고, 폭넓고, 병렬화되는** 작업에만 쓴다. 공유 문맥이 많거나 단계가 강하게 얽힌 일은 단일 흐름이나 Subagent가 낫다.

서브에이전트는 **깨끗한 컨텍스트**에서 일하고, 조정자(Orchestrator/리드)에게는 전체 작업 내역이 아니라 **압축 요약(대략 1~2천 토큰: 핵심 결정·결과·산출물 경로)** 만 돌려주게 한다. 원자료와 중간 과정은 `artifacts/` 파일에 남기고 요약만 올려, 조정자 컨텍스트가 오염되지 않게 한다. 이것이 멀티에이전트가 단일 컨텍스트 한계를 넘는 핵심 메커니즘이다.

모델은 위 "Agent 모델 선택" 루브릭으로 역할에 맞춰 고른다. Agent Team에서는 teammate 기본 모델 설정이나 팀 생성 지시가 실행 모델에 영향을 줄 수 있으므로, Orchestrator와 벤치마크 기록에는 실제 모델 정책을 따로 적는다.

## Agent 파일 구조

```md
---
name: example-agent
description: 이 Agent가 언제 호출되어야 하는지 구체적으로 쓴다.
tools: Read, Grep, Glob
model: sonnet   # 필수. 역할별로 고른다 → haiku=정적·추출·분류 / sonnet=분석·리뷰·작성(기본) / opus=설계·상충해소·구조판단 / fable=장시간 자율. 아래 "Agent 모델 선택" 루브릭 참조
---

당신은 {역할 이름}입니다.

## 책임

- 이 Agent가 맡는 일을 적는다.

## 입력

- 어떤 자료와 조건을 받아야 하는지 적는다.

## 출력

- 다음 단계가 바로 사용할 수 있는 산출물 형식을 적는다.
- 산출물 파일 경로를 함께 적는다.

## 작업 방식

1. 목적과 산출물을 확인한다.
2. 필요한 자료를 읽는다.
3. 근거와 추측을 구분한다.
4. 정해진 형식으로 결과를 남긴다.

## 팀 통신 프로토콜

- 메시지 수신: {누구에게 어떤 요청을 받는가}
- 메시지 발신: {어떤 발견, 질문, 충돌, 완료 알림을 누구에게 보내는가}
- Task 처리: 공유 작업 목록에서 {어떤 유형의 Task}를 맡는다.
- 파일 산출물: 결과는 반드시 `artifacts/{phase}-{role}-{artifact}.md`에 남긴다.
- 차단 조건: {어떤 상황에서 멈추고 Orchestrator 또는 사람에게 확인하는가}

## 하지 말아야 할 일

- 역할 밖의 결정을 확정하지 않는다.
- 확인하지 않은 사실을 단정하지 않는다.
```

기본 Agent 템플릿은 `name`, `description`, `tools`, `model`(역할별 루브릭)을 포함한다. `disallowedTools`, `permissionMode`, `maxTurns`, `effort`, `isolation`, `mcpServers`, `color` 같은 추가 frontmatter는 필요할 때만 검토한다.

특정 작업 매뉴얼을 반드시 따라야 하면 Agent 본문과 Orchestrator의 Task 설명에 적는다. `skills` frontmatter는 직접 Subagent로 호출할 때의 보조 설정으로만 사용하고, Agent Team teammate 실행의 필수 전달 수단으로 보지 않는다.

## 팀 패턴

| 패턴 | 쓰기 좋은 경우 | Agent Team 적용 방식 |
| --- | --- | --- |
| Pipeline | 순서가 분명하고 앞 단계 결과가 다음 단계 재료가 된다 | 각 단계 산출물 파일과 완료 기준을 Task로 등록한다 |
| Fan-out/Fan-in | 여러 자료를 나눠 읽고 합친다 | 병렬 조사 Task를 나누고 발견 공유와 중복 제거 메시지 규칙을 둔다 |
| Expert Pool | 요청마다 필요한 전문성이 다르다 | 필요한 전문가만 팀원으로 만들고 충돌 의견은 Orchestrator가 통합한다 |
| Producer-Reviewer | 초안과 검토를 분리해야 한다 | 작성자와 검토자가 메시지로 수정 요청과 반영 결과를 주고받는다 |
| Supervisor | 요청마다 담당자가 달라진다 | Orchestrator가 Task 상태를 보고 재배정하거나 누락 작업을 추가한다 |
| Hierarchical Delegation | 큰 프로젝트를 여러 하위 프로젝트로 쪼갠다 | 중첩 팀 대신 단계별 팀 재구성과 파일 handoff를 사용한다 |
| Evaluator-Optimizer Loop | 합격 기준이 분명하고 통과까지 다듬어야 한다 | 종료 계약(통과·횟수 상한·수렴·예산)과 best 롤백을 둔다 (아래 반복·합의·자기비판 패턴) |
| Voting/Self-Consistency | 같은 판정의 신뢰도를 높이거나 환각을 줄여야 한다 | 같은 작업을 N회 돌려 다수결·best-of-N으로 고른다 |
| Reflection | 별도 검토자를 둘 만큼 무겁지 않다 | 한 Agent가 자기 출력을 비판·수정하되 외부 기준과 함께 쓴다 |

## 팀 크기 가이드

| 작업 규모 | 권장 팀원 수 | 팀원당 Task 수 |
| --- | --- | --- |
| 소규모 | 2-3명 | 3-5개 |
| 중규모 | 3-5명 | 4-6개 |
| 대규모 | 5-7명 | 4-5개 |

팀원이 많을수록 조율 비용이 커진다. 같은 자료를 모두가 계속 읽어야 한다면 팀원을 늘리기보다 Phase를 나누거나 Agent 책임을 합친다.

## 생성자-평가자 분리

긴 작업에서 가장 흔한 실패는 결과물을 만든 Agent가 자기 결과를 스스로 합격으로 판정하는 것이다. 모델은 자기 산출물을 과대평가하는 경향(자기평가 편향)이 있어, 생성과 평가를 한 역할에 두면 검증이 형식만 남는다. 그래서 위험도가 있는 하네스에서는 만드는 역할(Generator)과 평가하는 역할(Evaluator)을 분리한다.

| 구분 | Generator | Evaluator |
| --- | --- | --- |
| 목적 | 산출물을 만든다 | 산출물이 기준을 넘는지 회의적으로 판정한다 |
| 보는 것 | 작업 맥락 전체 | 결과물과 합격 기준만 (작성 맥락은 최소화) |
| 튜닝 방향 | 충실하게 완성 | 의심을 기본값으로, 애매하면 불합격 |
| tools | 작성에 필요한 범위 | 읽기·검증 중심으로 좁힘 |

Evaluator를 회의적으로 튜닝하는 방법:

1. 합격·불합격 판정 예시를 2-3개씩 본문에 넣어 기준을 고정한다. "이 정도면 통과", "이건 왜 탈락"을 구체적으로 보여준다.
2. 채점 기준을 항목으로 분해하고, 품질이 약한 항목에 가중치를 둔다. 모든 항목을 같은 무게로 보면 약점이 평균에 묻힌다.
3. "애매하면 불합격" 같은 기본값을 명시해 과신을 막는다.
4. Evaluator는 산출물의 해당 줄·수치·테스트 결과 같은 증거로 판정하고, "괜찮아 보인다"는 표현을 금지한다.

생성과 평가가 직접 대화로 얽히면 평가가 무뎌진다. 가능하면 Generator의 결과를 `artifacts/` 파일로 남기고, Evaluator는 그 파일과 합격 기준만 읽어 판정하게 한다. 명시적 체크포인트(파일 교환)가 실시간 대화보다 더 엄격한 검증을 만든다.

평가자는 분리만으로 충분하지 않다. 순수 자기평가는 외부 신호 없이는 오히려 품질을 떨어뜨릴 수 있고, 평가자도 자기 출력을 선호하거나 더 길거나 앞선 답을 선호하는 편향을 가진다. 그래서 Evaluator의 판정은 가능하면 외부 신호에 묶는다: 테스트 실행 결과, 체크리스트, 결함 주입 회귀 세트, 수치·인용 대조, 사람 승인. 외부 기준이 마땅치 않은 주관 영역(문체·디자인)에서는 평가자를 생성자와 다른 관점으로 분리하고, 위치·장황함·자기선호 편향을 줄이도록 판정 기준을 미리 고정한다.

## 반복·합의·자기비판 패턴

순차·병렬·감독 같은 구조 패턴 외에, 결과 품질을 끌어올리는 "동적 제어" 패턴이 있다. 위험도가 높거나 합격 기준이 분명한 작업에서 골라 쓴다.

### 반복 검증 루프 (Evaluator-Optimizer)

Producer-Reviewer를 "한 번 검토"가 아니라 통과까지 반복하는 형태다. 생성 → 평가 → 미달이면 수정 → 다시 평가. 명확한 합격 기준이 있을 때 효과가 크다. 다만 무한 반복은 토큰 폭주, 에러 누적, 자기교정 역행(고칠수록 나빠짐)을 부르므로 반드시 종료 계약을 둔다.

| 출구 | 조건 | 처리 |
| --- | --- | --- |
| 통과 | 평가자가 합격 기준 충족 판정 | 정상 종료 |
| 횟수 상한 | 정한 N회 도달 (기본 2~3회) | 자동 통과 금지 → `미검증` 또는 `사람 승인 필요`로 에스컬레이션 |
| 수렴 정체 | 직전 라운드 대비 개선이 임계값 미만 | 더 못 올린다고 보고 멈춤 |
| 예산 상한 | 토큰·시간 한도 초과 | 현재 best와 사유를 보고하고 멈춤 |

추가 가드:

- best 보존과 롤백: 각 라운드 결과를 보존하고 "지금까지 최고"를 따로 둔다. 어떤 라운드가 점수를 떨어뜨리면 그 수정을 버리고 best로 되돌린다. 단조 개선을 가정하지 않는다.
- 외부 신호 결합: 매 라운드 평가를 테스트·체크리스트·결함 주입 회귀 세트에 묶어 자기평가 편향이 라운드마다 증폭되지 않게 한다.
- 두께 연동: 저위험은 루프를 생략하고 단발 검증, 고위험만 루프 + 미통과 시 사람 승인.

### 투표와 자기일관성 (Voting / Self-Consistency)

같은 작업을 여러 번 독립 실행해 다수결이나 best-of-N으로 고른다. 서로 다른 하위작업을 나누는 Fan-out/Fan-in과 다르다. 여기서는 같은 작업을 N회 돌린다. 분류·판정·환각 저감처럼 신뢰도가 중요한 단계에서 쓴다. 비용이 N배이므로 핵심 판정에만 제한적으로 쓴다.

### Reflection (자기비판)

별도 검토 Agent를 띄우지 않고, 한 Agent가 자기 출력을 비판하고 고치는 가벼운 루프다. Producer-Reviewer보다 싸지만, 외부 신호 없는 순수 자기비판은 한계가 있으므로 위험도가 낮거나 외부 기준(테스트·체크리스트)을 함께 줄 수 있을 때 쓴다.

### Routing (입력 분류 후 전문가 선택)

Expert Pool을 쓸 때 "어떤 전문가를 부를지"를 즉흥 판단하지 말고, 입력을 먼저 분류하는 단계를 1급으로 둔다. 분류 기준과 어디에도 안 맞을 때의 기본 경로를 명시하면 라우팅이 안정된다.

## Agent Team 안티패턴

Skill 작성 자체(구조·description·본문·progressive disclosure)는 `skill-authoring-guide.md`를 정본으로 따른다. 여기서는 팀 설계에서 피해야 할 구조만 둔다.

- Agent 파일만 여러 개 만들고 `TeamCreate` 흐름이 없는 구조
- 메시지로만 중요한 결정을 주고받고 파일 산출물을 남기지 않는 구조
- 모든 Agent가 모든 파일을 동시에 고치는 구조
- Orchestrator가 모든 일을 직접 하고 Agent Team을 장식처럼 두는 구조
