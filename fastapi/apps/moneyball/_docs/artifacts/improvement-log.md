**되살릴 근거**: 제거한 것은 전부 다른 섹션에 동일 내용이 남아 있는 순수 중복. 모델이 약해지거나 규칙이 흐려지면 git 히스토리(이 커밋 직전)에서 복원 가능.

## 2026-06-21 — ADR 도입 (이 항목 자체가 ADR 형식 예시)

- **결정**: harness-lab에 경량 ADR을 도입하되 변경 이력(changelog)과 **병행**한다. 큰 구조 결정만 ADR, 자잘한 변경은 changelog.
- **버린 대안**: (a) changelog를 ADR로 전면 교체 — 모든 자잘한 변경에 대안·트레이드오프를 쓰게 돼 과설계, 방금 한 덜어내기와 역행. (b) 정식 ADR 시스템(별도 `adr/` 디렉터리·번호·상태) — 무게 과다.
- **가정·트레이드오프**: "덜어내기 진단은 각 조각이 인코딩한 가정을 알아야 가능하다"는 가정 위에 섬. ADR이 그 가정을 보존. 트레이드오프: 큰 결정마다 4줄 기록 비용.
- **재검토 조건**: ADR이 실제로 안 쓰이거나(덜어내기 진단이 ADR을 참조 안 함) 작성 부담이 가치를 넘으면, 설계 카드 칸만 남기고 별도 ADR 기록은 줄인다.

## 2026-06-21 — ADR: Phase 1을 "일상 언어로 풀어보기" → "도메인 분석"으로 교체

- **결정**: Phase 1을 도메인 분석으로 바꾼다. "일상 언어로 풀어보기"는 독립 Phase가 아니라 도메인 분석의 한 기법("사람이 한다면 어떤 순서로?")으로 흡수한다. Phase 2~7 번호·내용은 그대로 둔다.
- **버린 대안**: (a) 도메인 분석을 새 Phase로 추가하고 Agent 설계+Skill 설계를 병합 — granularity 손실, 불필요. (b) 도메인 분석을 Phase 0에 sub-bullet으로 흡수 — 가장 중요한 분석인데 묻힘. (c) Phase 0~7 불변식을 깨고 Phase 8 추가 — 일관성 검사기·전 참조 문서에 광범위 ripple.
- **가정·트레이드오프**: "역할 분리·승인 지점·적합성 판정이 모두 도메인 이해에서 나온다"는 가정. 일상언어 풀이의 가치는 도메인 분석 안 기법 + 기존 원칙으로 보존되어 손실 없음. 트레이드오프: 교육용 "일상언어" 단계의 독립 가시성은 약간 줄어듦.
- **재검토 조건**: 사용자가 일상언어 풀이를 다시 독립 단계로 보고 싶어 하거나, 도메인 분석이 너무 무거워 초보자가 막히면 재분리한다.
- **효과**: harness-lab Phase가 원본 harness(0감사·1도메인·2아키텍처...)와 거의 1:1 정렬(차이는 harness-lab 고유의 Phase 2 산출물 정의뿐). 가장 중요한 빈 구석이던 도메인 분석이 제 위치(Phase 1)로 들어옴.

## 2026-06-23 — ADR: Agent 도구 권한 ≠ 폐지, "생산자 Write 보장 + 저장 계약"으로 해결

- **상태**: 채택
- **맥락**: 사용자가 실제 하네스(사업계획서)를 돌려 tools 권한 버그 2건이 드러남. (B1) `readiness-reviewer`에 `Write`가 없어 자기 판정 파일(`readiness.md`)을 못 씀 — 임시로 Orchestrator가 파일화해 메움. (B2) `prd-author`가 `Write`가 있는데도 `analysis.md` 저장 실패를 헛보고하고 본문 요약으로 대체. 사용자가 "차라리 tools를 없애는(상속) 게 낫나?"를 질문.
- **결정**: tools 폐지(생략=전체 상속)는 채택하지 않는다. 대신 두 근본 원인을 분리해 고친다 — (1) **산출물을 파일로 남기는 모든 역할은 `Write`가 필요**하다(검토·QA 포함). "최소 권한"을 "쓰기 금지"가 아니라 "쓰기 대상을 자기 산출물로 한정 + 대상 수정 권한 `Edit`은 분리"로 재정의. (2) 위임/본문에 **저장 계약**(지정 경로 저장 → 실패 시 재시도 → 차단 보고, 저장한 척 본문 요약 대체 금지)을 명시. (3) **도구↔산출물 정합** 프리플라이트를 테스트 체크리스트에 추가.
- **버린 대안**: (a) `tools` 전면 생략(상속) — B1은 우연히 가려지나 B2는 그대로이고, reviewer가 `Edit`까지 얻어 검증 대상을 직접 고침 → 생성자·평가자 분리(이 스킬의 핵심 안전 원칙)가 깨짐. (b) reviewer를 읽기 전용 유지하고 Orchestrator가 항상 판정 파일을 대신 씀 — 검토자의 산출물 책임이 Orchestrator로 새어 역할 경계가 흐려지고 매 하네스에 보일러플레이트.
- **가정·트레이드오프**: B2는 권한 문제가 아니라 출력 계약 미준수라는 진단 위에 섬. 트레이드오프: tools 표가 약간 복잡해지고(생산자 Write 구분), 위임마다 저장 계약 한 줄 비용.
- **재검토 조건**: 모델이 저장 실패를 스스로 정확히 보고·복구하게 되어 저장 계약이 불필요해지거나, per-agent 도구 모델이 바뀌면(예: 산출물 전용 쓰기 스코프 기본 제공) 재검토.
- **반영 위치**: claude `agent-design.md`(tools 표 Reviewer 행 + 생산자 Write 원칙), `testing-qa-evolution.md`(체크리스트 정합 항목), `orchestrator-design.md`(Task 저장 계약), `SKILL.md`(품질 기준). codex는 TOML이 세션 도구를 상속하므로 A 대신 `agent-skill-design.md`·TOML 템플릿·`failure-maintenance.md` Drift 표·`SKILL.md`에 저장 계약(C)과 저장 정합(B)을 반영(Claude 이식 시 Write 확인 주석 포함).
